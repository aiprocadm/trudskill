# Нагрузочное тестирование (k6) — Phase 11

Лёгкий, **безопасный для данных** (только чтение + liveness) нагрузочный харнесс для пилотного выката.

## Контекст по NFR

На v1 **числовые SLA не согласованы** (см. [`docs/NFR_LAUNCH_V1.md`](../../docs/NFR_LAUNCH_V1.md)): uptime %, RPO/RTO и профиль одновременной нагрузки заказчиком пока не заданы. Поэтому по умолчанию скрипт работает как **smoke/ramp с мягким порогом ошибок** (`errors < 5%`) и **без** порога по задержке. Это осознанно: до появления чисел нагрузочный прогон служит проверкой работоспособности, а не приёмкой SLA.

## Предпосылки

- Установленный [k6](https://k6.io/docs/get-started/installation/).
- Развёрнутый стенд (staging/прод), желательно с засеянным пилотным тенантом.
- Для аутентифицированных чтений — действующий JWT (`Bearer`) и совпадающий `tenant_id` (его требует `TenantGuard`).

## Запуск

```bash
# 1) Smoke без авторизации — только health/ready
k6 run -e BASE_URL=https://app.example.ru infra/load/k6-smoke.js

# 2) + представительное аутентифицированное чтение (workspace/summary)
k6 run -e BASE_URL=https://app.example.ru -e TENANT_ID=tenant_demo -e TOKEN=<jwt> \
  infra/load/k6-smoke.js

# 3) Рамп + согласованный SLA (когда заказчик дал числа)
k6 run -e BASE_URL=https://app.example.ru -e TENANT_ID=tenant_demo -e TOKEN=<jwt> \
  -e VUS=50 -e DURATION=5m -e THRESHOLD_P95_MS=800 -e THRESHOLD_ERROR_RATE=0.01 \
  infra/load/k6-smoke.js
```

## Параметры (env)

| Переменная             | Default                 | Назначение                                           |
| ---------------------- | ----------------------- | ---------------------------------------------------- |
| `BASE_URL`             | `http://localhost:3001` | Базовый URL стенда (без хвостового `/`)              |
| `API_PREFIX`           | `/api/v1`               | Префикс API                                          |
| `TOKEN`                | —                       | JWT для аутентифицированных чтений (опц.)            |
| `TENANT_ID`            | —                       | Должен совпадать с `tenant_id` в JWT (`TenantGuard`) |
| `VUS`                  | `10`                    | Целевое число виртуальных пользователей              |
| `DURATION`             | `30s`                   | Длительность плато нагрузки                          |
| `THRESHOLD_P95_MS`     | — (выкл.)               | Порог p95 по задержке; включается только если задан  |
| `THRESHOLD_ERROR_RATE` | `0.05`                  | Допустимая доля ошибок                               |

## После прогона

1. Если заказчик согласовал числовые NFR — выставьте пороги (`THRESHOLD_*`), повторите прогон с целевым `VUS`/`DURATION`.
2. Зафиксируйте результаты (p95, доля ошибок, профиль) и ссылку на прогон в [`docs/NFR_LAUNCH_V1.md`](../../docs/NFR_LAUNCH_V1.md) (раздел «После получения числовых NFR»).
3. Целевой ориентир из роадмапа — **1000 учеников/мес.** (см. Phase 11 в мастер-роадмапе); это ёмкость, а не одновременность — конкретный профиль concurrency согласуется с заказчиком.

## Безопасность

Скрипт выполняет **только** `GET` (health + `workspace/summary`) — никаких мутаций. Не коммитьте реальные `TOKEN` в репозиторий; передавайте через `-e` или CI-секрет.
