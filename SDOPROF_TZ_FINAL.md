## 1. Назначение системы

СДО Проф — корпоративная платформа управления обучением, аттестациями, документами и аналитикой для организаций с внутренним и внешним контуром обучения.

Система должна:

- обеспечивать полный цикл обучения: от назначения до подтверждения результата документом;
- поддерживать корпоративное, обязательное, адаптационное, квалификационное, compliance-обучение, охрану труда, промышленную безопасность;
- поддерживать дистанционный, смешанный и очный форматы с фиксацией результатов;
- обеспечивать работу с внутренними сотрудниками и, при включении режима, внешними слушателями/партнерами.

Целевая аудитория:

- крупные и средние организации;
- учебные центры в составе компаний;
- образовательные подразделения с требованиями аудита и юридически значимых следов.

## 2. Цели и границы проекта

### 2.1 Цели MVP

Система должна обеспечить:

- управление пользователями, ролями и оргструктурой;
- каталог курсов, базовый конструктор, публикацию и назначение;
- прохождение обучения, прогресс, тесты, задания;
- генерацию документов/сертификатов;
- базовую отчетность;
- аудит критичных действий и безопасный доступ.

### 2.2 Цели полной версии

Система должна дополнительно обеспечить:

- компетентностную модель и траектории;
- развитую аналитику и конструктор отчетов;
- интеграции с внешними системами;
- электронные заявки и согласования (если подтверждено);
- электронное подписание и legal log (если подтверждено);
- AI-функции персонализации и ассистентов.

### 2.3 Границы

В систему входят:

- backend API, frontend web, БД, файловое хранилище, очереди, аудит, отчеты.

Не входят (по умолчанию):

- разработка внешних HRM/ERP систем;
- разработка внешней КЭП/ЭДО платформы;
- производство учебного контента вне LMS.

### 2.4 Ограничения и зависимости

- Требует уточнения: формальные отраслевые/юридические регламенты.
- Требует уточнения: перечень обязательных внешних интеграций по приоритетам.
- Требует уточнения: требования к on-prem / cloud модели размещения.

### 2.5 Рекомендуемый состав MVP (пилот)

Ниже — **ориентир** для минимально жизнеспособного продукта. Окончательный список фиксируется с заказчиком (протокол / приложение к договору). Детальная связка с бэклогом — [§41](#41-backlog-укрупнённый).

| Область     | MVP (ориентир)                                                       | Post-MVP / вне pilot без решения                                              |
| ----------- | -------------------------------------------------------------------- | ----------------------------------------------------------------------------- |
| Доступ      | IAM, RBAC, tenant isolation, логин/сессии, аудит критичных действий  | SSO/LDAP как **Should**, до подтверждения протокола                           |
| Курсы       | Каталог, модули/материалы, публикация, базовые правила прохождения   | SCORM/xAPI как расширение                                                     |
| Назначения  | Одиночные и **массовые** назначения, журнал ошибок массовых операций | Сложные потоки, лист ожидания, согласования участия                           |
| Прохождение | Прогресс по курсу группы/назначения, дедлайны на уровне полей данных | Полный календарь активностей, мобильные ограничения для экзаменов по политике |
| Оценивание  | Тесты (банк, попытки, scoring), задания и ревью с жизненным циклом   | Апелляции, расширенные типы вопросов по необходимости                         |
| Документы   | Шаблоны, генерация, реестр, выдача слушателю по правилам завершения  | Юридически значимое НЭП/КЭП, legal log без юр. модели                         |
| Отчёты      | KPI (completion / pass rate и др. по согласованию), фильтры, экспорт | Конструктор отчётов, рассылка по расписанию                                   |
| Интеграции  | REST API, webhooks при необходимости пилота                          | HRM/календарь/видео — после приоритизации                                     |

**Явно не включать в MVP без отдельного решения:** многоступенчатый документооборот заявок; юридически значимое подписание; расширенные AI-сценарии (см. [§22](#22-ai-функции-и-современные-edtech-возможности)).

Пересечение с «полной версией»: цели полной версии — п. **2.2** выше; границы контура — п. **2.3** выше.

## 3. Пользователи, роли и права доступа

Система должна реализовать RBAC с tenant- и scope-ограничениями.

Роли:

- супер-администратор платформы;
- администратор организации;
- администратор обучения;
- методолог;
- автор курса;
- преподаватель;
- тьютор;
- эксперт/проверяющий;
- руководитель подразделения;
- HR/L&D;
- слушатель;
- внешний слушатель;
- аудитор;
- техподдержка.

Для каждой роли система должна задавать:

- доступные разделы;
- CRUD-операции по сущностям;
- ограничения по организации/подразделению/курсу/документу;
- запрет эскалации прав;
- аудит изменений прав.

Типовые сценарии:

- слушатель: просмотр назначений, прохождение, сдача заданий, получение документов;
- руководитель: назначение/контроль подчиненных;
- эксперт: проверка и оценка submissions;
- администратор: управление структурами, пользователями, курсами, назначениями.

## 4. Организационная структура

Система должна поддерживать:

- организации, филиалы, подразделения, должности;
- группы и потоки обучения;
- руководителей и матричные связи (при включении);
- внешние контрагенты/слушатели (при включении);
- учебные классы и проектные команды.

## 5. Пользователи и профили

Система должна:

- создавать пользователя вручную, массово, через приглашение;
- поддерживать импорт пользователей и оргструктуры;
- поддерживать блокировку, архивирование, восстановление;
- хранить профиль, учебную историю, компетенции, сертификаты, документы;
- поддерживать персональные настройки уведомлений, языка, часового пояса.

Требует уточнения:

- обязательность LDAP/AD/SSO-синхронизации на MVP.

## 6. Авторизация и безопасность доступа

Система должна:

- поддерживать логин/пароль и восстановление;
- поддерживать сессии, отзыв сессий, аудит входов;
- поддерживать защиту от brute-force;
- поддерживать tenant isolation;
- поддерживать 2FA (Should), SSO/OIDC/SAML (Should, по интеграционному плану);
- поддерживать ограничения доступа по IP (Could).

## 7. Каталог курсов

Система должна поддерживать:

- категории, теги, поиск, фильтры, сортировку;
- витрину публичных/закрытых курсов;
- курсы по назначению, по заявке, обязательные, рекомендованные;
- архив курсов;
- версии курсов.

Карточка курса должна содержать:

- название, описание, цели, программу, длительность, формат;
- целевую аудиторию, требования, автора/преподавателя;
- правила прохождения, дедлайны, проходной балл;
- сведения о документе по итогам, доступность, язык, статус.

## 8. Конструктор курсов

Система должна поддерживать:

- модульную структуру: модули/уроки/темы;
- типы контента: текст, видео, аудио, презентации, PDF, файлы, ссылки, HTML;
- оценочные элементы: тесты, задания, опросы, чек-листы;
- правила разблокировки и prerequisite;
- попытки и проходные пороги.

Система должна поддерживать жизненный цикл:

- черновик, предпросмотр, публикация, модерация;
- версионирование, копирование, импорт/экспорт;
- библиотеку переиспользуемого контента.

Поддержка стандартов:

- MVP: HTML5/PDF/video/file;
- Post-MVP: SCORM 1.2/2004;
- Перспектива: xAPI/cmi5/LTI.

## 9. Учебные траектории

Система должна поддерживать:

- индивидуальные и групповые траектории;
- обязательные/рекомендательные маршруты;
- траектории по должности и компетенциям;
- правила перехода между этапами;
- дедлайны и контроль прогресса;
- автоназначение следующего обучения.

## 10. Компетентностная модель

Система должна поддерживать:

- справочник компетенций и уровней;
- профили должностей и матрицы;
- связь компетенций с курсами;
- диагностику и gap-анализ;
- ИПР;
- подтверждение компетенций и переаттестацию;
- отчеты по компетенциям.

## 11. Назначение и запись на обучение

Система должна поддерживать:

- self-enroll, enroll по заявке, назначение администратором/руководителем;
- автоназначение по правилам (должность/подразделение/событие);
- массовое назначение, повторное назначение, отзыв;
- перенос сроков, продление доступа;
- потоки, лимиты мест, лист ожидания;
- согласование участия (если включено).

## 12. Прохождение обучения слушателем

Система должна предоставить слушателю:

- личный кабинет и список назначений;
- прогресс и дедлайны;
- возобновление с точки остановки;
- прохождение тестов и заданий;
- доступ к сертификатам/документам;
- историю обучения и рекомендации;
- календарь активности;
- мобильный web-доступ.

## 13. Тестирование и контроль знаний

Система должна поддерживать:

- банк вопросов и пулы;
- типы: single/multi choice, текст, число, соответствие, сортировка, эссе;
- таймер, попытки, пороги, рандомизацию;
- ручную проверку эссе/кейсов;
- разбор ошибок;
- апелляцию (Should);
- отчеты по результатам.

## 14. Экзамены, аттестации и переаттестации

Система должна поддерживать:

- экзаменационные сессии и расписание;
- допуски, попытки, результаты, пересдачи;
- протоколы и документы;
- периодическую переаттестацию и автоназначение.

## 15. Практические задания

Система должна поддерживать:

- выдачу задания и дедлайны;
- текстовый ответ и загрузку файлов;
- критерии и rubrics;
- назначение проверяющего;
- комментарии и возврат на доработку;
- версии ответа;
- оценку и журнал проверки;
- апелляцию (Should).

## 16. Вебинары, очные мероприятия и blended learning

Система должна поддерживать:

- расписание мероприятий, регистрацию и посещаемость;
- онлайн-встречи через интеграции (Should);
- материалы и пост-задания;
- перенос/отмену;
- календарные приглашения.

Требует уточнения:

- обязательность QR-регистрации на MVP.

## 17. Документы, сертификаты и удостоверения

Система должна поддерживать:

- шаблоны документов;
- авто/ручную генерацию;
- нумерацию и правила нумерации;
- QR-проверку подлинности;
- реестр документов;
- аннулирование и перевыпуск;
- хранение, экспорт и печать.

## 18. Электронные заявки и документооборот

Статус: Требует уточнения по исходному ТЗ.

Если подтверждено, система должна поддерживать:

- типы заявок, статусы, маршруты согласования;
- вложения, комментарии, журнал изменений;
- связь заявок с обучением и документами.

До подтверждения — модуль относится к перспективным.

## 19. Электронное подписание / НЭП / legal log

Статус: Частично подтверждено текущей кодовой базой (e-sign/ legal-log модуль).

Система должна поддерживать:

- процессы подписания и участников;
- статусы подписи и события;
- неизменяемость финальных подписанных артефактов;
- legal log и доказательную базу;
- отказ и повторный запуск процесса.

Требует уточнения:

- юридическая модель НЭП/КЭП и регламент хранения.

## 20. Уведомления и коммуникации

Система должна поддерживать:

- email и внутренние уведомления;
- push (Should);
- SMS/мессенджеры (Could);
- шаблоны по событиям;
- напоминания о дедлайнах и эскалации;
- digest-уведомления;
- журнал доставки.

## 21. Социальное и совместное обучение

MVP (Should):

- комментарии и вопросы преподавателю.

Post-MVP:

- форумы, peer-review, учебные группы.

Перспектива:

- рейтинги, бейджи, лидерборды, сообщества практики.

## 22. AI-функции и современные EdTech-возможности

### MVP (ограниченно)

- semantic search по контенту (если включено);
- базовые рекомендации курсов по ролям/траекториям.

### Post-MVP

- AI-помощник слушателя;
- AI-помощник автора курса;
- черновики тестов и резюме материалов;
- риск-незавершения и gap-анализ.

### Перспектива

- ассистенты администраторов/методологов;
- полуавтоматическая проверка текстовых ответов с human-in-the-loop.

Обязательные ограничения:

- маркировка AI-generated контента;
- запрет юридически значимых автоматических решений без участия человека;
- соблюдение требований защиты персональных данных.

## 23. Аналитика и отчётность

Система должна предоставлять:

- dashboards для админа, руководителя, L&D, аудитора;
- отчеты по пользователям, курсам, прогрессу, тестам, аттестациям, компетенциям, документам;
- drill-down и фильтры;
- экспорт CSV/XLSX/PDF;
- плановые и рассылаемые отчеты (Should).

Ключевые метрики:

- completion rate;
- pass rate;
- average score;

## 25. Интеграции

- HRM/HCM (Should);
- LDAP/AD (Should);
- видеоконференции/календари (Should);

Для каждой интеграции должны быть заданы:

- формат;
- обработка ошибок и retry;
- аудит и безопасность.

## 26. API

Система должна:

- использовать REST API;
- поддерживать версионирование;
- публиковать OpenAPI;
- поддерживать auth, rate limiting;
- поддерживать pagination/filtering/sorting;

- использовать единый error model;
- прокидывать correlation ID;
- поддерживать webhooks;
- сохранять backward compatibility в пределах версии.

## 27. Импорт и экспорт данных

## 45. Рекомендуемый порядок разработки и readiness

Рекомендуемая последовательность (согласуется с [§40](#40-mvp-и-этапы-реализации)):

1. Этапы **0–1**: требования, доменная модель, ядро доступа, аудит.
2. Этапы **2–5**: каталог и курсы, назначения и прогресс, тестирование, задания.
3. Этап **6**: документы и сертификаты.
4. Этап **8**: отчётность и KPI (минимум для пилота).
5. Этап **9+**: интеграции и расширения по приоритету.
6. Этапы **13–14**: доведение тестового контура и production readiness (observability, backup, rollback, runbooks).

**Готовность к контракту «фиксированный MVP»:** зафиксированы перечень Must из [§41](#41-backlog-укрупнённый), числовые или явно отклонённые NFR, перечень интеграций и юридический минимум для документов и подписания.

**Готовность к передаче в эксплуатацию:** выполнены критерии [§39](#39-acceptance-criteria), актуальны runbooks и миграции БД на целевых окружениях ([§37](#37-devops-и-окружения)).

## 46. Приложение А. Глоссарий терминов ТЗ

Единые определения для разделов [§2–§41](#2-цели-и-границы-проекта) и для [docs/TZ_MVP_TRACEABILITY.md](docs/TZ_MVP_TRACEABILITY.md).

| Термин                                    | Краткое значение в контексте СДО Проф                                                                                                                                              |
| ----------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Tenant (арендатор)**                    | Логически изолированный контур данных организации: пользователи, курсы, назначения и т.д. не смешиваются между арендаторами без явных полномочий.                                  |
| **RBAC**                                  | Модель «роль → набор прав (permissions)»; доступ к объектам ограничен ролью и областью действия (оргструктура, курс и т.д.).                                                       |
| **Пилот / MVP**                           | Согласованный минимум функций первого промышленного внедрения; см. [§2.5](#25-рекомендуемый-состав-mvp-пилот) и backlog [§41](#41-backlog-укрупнённый).                            |
| **Enrollment (зачисление)**               | Связка «слушатель (или представитель организации) — курс/группа» с датами, статусом и правилами прохождения.                                                                       |
| **Группа обучения**                       | Учебная группа в рамках tenant; к ней привязываются зачисления и привязка курсов (**group_courses** в реализации).                                                                 |
| **Прогресс**                              | Зафиксированное выполнение материалов/этапов в рамках зачисления; не должен «утекать» между несогласованными курсами.                                                              |
| **Массовые назначения**                   | операция создания множества зачислений за один запрос; может выполняться синхронно или через очередь при больших объёмах (**BL-003**).                                             |
| **Org unit / подразделение**              | Узел оргструктуры; может использоваться для выбора аудитории при назначениях при согласованной модели данных.                                                                      |
| **Сертификат / сгенерированный документ** | Документ по шаблону после выполнения правил завершения обучения; ведётся в реестре (**BL-007**).                                                                                   |
| **Legal log / ЭП**                        | Неизменяемые записи и электронная подпись юридически значимых действий; в пилоте обычно вне минимального scope без юр. модели ([§19](#19-электронное-подписание--нэп--legal-log)). |
| **KPI / snapshot**                        | Свод показателей (например completion rate, pass rate) с фильтрами и при необходимости детализацией (**BL-008**).                                                                  |
| **IDOR**                                  | Класс уязвимостей: доступ к объекту по идентификатору без проверки полномочий на этот объект; предотвращение — **BL-010**.                                                         |
| **NFR**                                   | Нефункциональные требования: производительность, надёжность, RPO/RTO и т.д. ([§31](#31-нефункциональные-требования), [NFR_LAUNCH_V1](docs/NFR_LAUNCH_V1.md)).                      |
