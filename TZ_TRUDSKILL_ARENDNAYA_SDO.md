## 2. Целевая модель продукта

### 2.1 Два режима работы одной платформы

- **Режим «свой УЦ»**: ты — первый и главный тенант. Всё, что строим, сразу проверяется на собственном учебном центре.
- **Режим «аренда»**: другой УЦ получает изолированное пространство: свои курсы (или копии из библиотеки платформы), свои слушатели, свои бланки документов, свой логотип и поддомен, свои комиссии и нумераторы. Одна кодовая база, одна БД, изоляция по `tenant_id` (+ опционально RLS вторым слоем, см. ФТ-D1.3).

### 2.2 Иерархия ролей (3 уровня)

| Уровень                | Роли                                                                     | Что делает                                                                                                   |
| ---------------------- | ------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------ |
| Платформа              | `platform_admin`, поддержка                                              | Создаёт/приостанавливает тенантов, тарифы, лимиты, библиотека курсов платформы, tenant health                |
| Учебный центр (тенант) | `tenant_admin`, методист, преподаватель/куратор, член комиссии, менеджер | Курсы, группы, слушатели, шаблоны документов, комиссии, выгрузки в реестры, продажи                          |
| Внешние                | заказчик (counterparty) и его представитель, `learner`                   | Портал заказчика: свои сотрудники, прогресс, документы, сроки. Слушатель: обучение, экзамены, свои документы |

Роли IAM уже существуют — задача не создать их заново, а довести permission-модель до 100% endpoints (gap A2) и добавить платформенный уровень там, где его нет.

### 2.3 Мини-глоссарий (для однозначности в задачах)

**Тенант/УЦ** — арендатор платформы. **Заказчик** — организация-клиент УЦ (counterparty). **Направление** — вид обучения (ОТ программа Б, первая помощь…). **Версия курса** — зафиксированная программа с часами; группа учится на конкретной версии. **Комиссия** — состав для проверки знаний (минимум 3 человека по ПП 2464). **Нумератор** — счётчик номеров документов с маской. **ПЭП** — простая электронная подпись (соглашение + фиксация действий). **НЭП** — усиленная неквалифицированная (модуль esign).

## 3. ЭПИК A — Рендер документов и конструктор шаблонов **[P0]**

### Почему такой подход

Твоя цель: администратор УЦ **сам, без программиста** создаёт шаблон протокола заседания комиссии или удостоверения — и его форму, и наполнение. Самый надёжный путь — **DOCX-шаблоны с плейсхолдерами**: админ делает бланк в привычном Word (пиксельная точность существующих бланков УЦ гарантирована), загружает в систему, система подставляет данные и отдаёт DOCX + PDF. Визуальный конструктор «с нуля в браузере» — дорогая и хрупкая альтернатива; оставляем её на перспективу. Важно: вся серверная обвязка под это в проекте **уже есть** — не хватает именно движка рендера и админского UX.

### Требования

**ФТ-A1. Движок рендера (worker)**

- A1.1. Реализовать зависимость `render` в `DocumentGenerationOrchestrator` (`apps/worker/src/document-pipeline.ts`): взять DOCX-файл активной версии шаблона из S3 → подставить переменные → сохранить результат в S3 → вернуть `fileId`.
- A1.2. Библиотека: **docxtemplater** (open-source, MIT-ядро, поддерживает циклы/условия, живое сообщество). Альтернатива carbone отклонена из-за лицензионных ограничений community-версии — финальное слово за тобой (см. Открытые вопросы, п. 2).
- A1.3. PDF-конвертация: контейнер **Gotenberg** (LibreOffice внутри) в `infra/docker-compose.yml`; worker шлёт DOCX → получает PDF. Хранить оба формата.
- A1.4. **Детерминированный снапшот** (закрывает gap C4): в момент генерации сохранять в `generated_documents` полный JSON подставленных данных. Повторный рендер того же документа обязан давать тот же результат; изменение данных слушателя задним числом не меняет уже выданный документ.
- A1.5. Ошибки рендера → существующий механизм `FailureDiagnostics` + понятное сообщение админу («в шаблоне неизвестная переменная {learner.fio_dativ}»).

**ФТ-A2. Язык шаблонов**

- A2.1. Плейсхолдеры вида `{learner.full_name}`, `{program.academic_hours}`, `{document.number}` — строго из каталога переменных (10 категорий уже зафиксированы CHECK-констрейнтом миграции `0032`: tenant, group, learner, counterparty, course, commission, document, program, enrollment, group_learners).
- A2.2. Циклы для таблиц: `{#group_learners} … {/group_learners}` — один протокол на группу со строками по слушателям (ФИО, должность, организация, результат, номер записи). Это главный сценарий протокола заседания комиссии.
- A2.3. Условные блоки: `{#if …}` для необязательных секций (например, строка «повторная проверка знаний»).
- A2.4. Расширить каталог переменных: `learner.snils`, `learner.position`, `document.number`, `document.issue_date_words` (дата прописью), `commission.chairman_*` / `commission.members[]`, `tenant.accreditation_number`, `tenant.license_number`. Склонения ФИО (дательный падеж для удостоверений) — в MVP как необязательные ручные поля профиля слушателя; интеграция морфологического сервиса — P2.

**ФТ-A3. Админ-UX шаблонов (страница «Документы → Шаблоны»)**

- A3.1. Загрузка `.docx` через существующий files-модуль (антивирусный гейт обязателен в проде, см. ФТ-G5).
- A3.2. После загрузки система **парсит плейсхолдеры** и показывает таблицу: найдено / соответствует каталогу / неизвестно (подсветить красным с подсказкой ближайшей переменной).
- A3.3. **Тестовый предпросмотр**: кнопка «Сгенерировать пример PDF» на демо-данных тенанта — админ сразу видит, как ляжет бланк.
- A3.4. Версионирование — сущности `TemplateVersionEntity` уже есть: новая загрузка = новая версия, активная переключается явно; выданные документы навсегда помнят свою версию.
- A3.5. Привязки шаблона к направлению/курсу/группе (`TemplateBindingEntity` есть) — довести UI выбора и приоритетов.
- A3.6. Стартовый набор при онбординге тенанта: типовые шаблоны «Протокол проверки знаний (ПП 2464)», «Удостоверение», «Приказ о создании комиссии», «Справка об обучении» — копируются из библиотеки платформы, дальше УЦ правит под себя.

**ФТ-A4. Нумераторы**

- A4.1. UI настройки per tenant per тип документа: маска (например `{YY}-ОТ-{SEQ:4}`), стартовое значение, период сброса (поле `NumberResetPeriod` уже есть: none/year/month).
- A4.2. Резервирование номера уже встроено в пайплайн (`reserveNumber`) — гарантировать отсутствие дыр/дублей при падении рендера (номер освобождается или помечается «аннулирован»).

**ФТ-A5. Массовая генерация**

- A5.1. Сценарий «закрыть группу»: одной операцией — протокол на группу (1 документ с таблицей) + удостоверение на каждого сдавшего (N документов) + запись в журнал выдачи (`/admin/issuance-journal` — маршрут есть, наполнить данными).
- A5.2. Выгрузка: ZIP всех PDF группы; отправка слушателям на email — после включения почты (ЭПИК F).
- A5.3. Статусы задач видны админу; частичный сбой (упал 1 из 25) — повторный запуск только упавших.

**ФТ-A6. Публичная проверка подлинности**

- A6.1. Страница `/verify/{qr_token}` (бэкенд `public-verify.controller.ts` уже есть): номер, тип документа, программа, даты, статус (действует / отозван) — **без лишних ПДн** (ФИО показывать частично: «Иванов И. И.»).
- A6.2. Защита от перебора токенов: rate limiting (см. ФТ-G2).

**ФТ-A7. Изображения и подписи**

- A7.1. Переменные-изображения: подпись руководителя, печать УЦ (хранятся в реквизитах тенанта) — опционально вставляются в шаблон. Юридический статус — факсимиле; для юридически значимой подписи связка с esign/НЭП (модуль есть).

**ФТ-A8. Комплаенс-валидатор протокола**

- A8.1. Для шаблонов типа `protocol` — мягкая проверка на обязательные поля из п. 92 ПП 2464 (наименование организации, номер и дата протокола, ФИО/должность, программа, результат, подписи членов комиссии, номер записи в реестре). Предупреждение, не блокировка.

**Критерий приёмки эпика:** администратор тенанта без участия разработчика загружает собственный бланк удостоверения и протокола, генерирует комплект на реальную группу из 20+ слушателей; PDF совпадает с бланком, номера сквозные без дублей, QR-код открывает страницу проверки; повторный рендер выдаёт идентичный файл.

## 4. ЭПИК B — Видео и учебный контент **[P0/P1]**

### Выбор архитектуры (нужно твоё решение)

|                    | Вариант 1: провайдер (Kinescope / VK Видео)           | Вариант 2: self-hosted (MinIO + ffmpeg + HLS)                |
| ------------------ | ----------------------------------------------------- | ------------------------------------------------------------ |
| Скорость внедрения | Быстро: загрузка через API, их плеер, их CDN и защита | Дольше: пайплайн транскодирования, плеер, подписанные ссылки |
| Деньги             | Ежемесячная плата, растёт с объёмом                   | Только своё железо/сторадж; выгоднее при аренде многим УЦ    |
| Контроль/152-ФЗ    | Зависимость от провайдера                             | Полный контроль, данные у себя                               |
| Риск               | Низкий                                                | Средний (нагрузка на отдачу видео)                           |

**Рекомендация:** сделать интерфейс `VideoProvider` (по образцу уже существующего webinar provider seam, миграция `0055`) с двумя реализациями. Стартовать с варианта 1 для скорости, self-hosted подключить к фазе аренды — тогда лимиты хранилища станут статьёй тарифа, а не внешним счётом.

### Требования

**ФТ-B1. Загрузка и хранение**

- B1.1. Загрузка видео методистом в урок (тип материала `'video'` уже есть в `mvp.types.ts`): через провайдера или напрямую в S3 (multipart, до 2–4 ГБ), статусы «загружается → обрабатывается → готово».
- B1.2. Self-hosted ветка: worker-задача транскодирования ffmpeg → HLS (3 качества: 480/720/1080), очередь RabbitMQ уже есть.
- B1.3. Лимит хранилища per tenant (счётчик занятого места) — фундамент тарифов аренды (ФТ-D4).

**ФТ-B2. Защищённое воспроизведение**

- B2.1. Плеер hls.js (или плеер провайдера): запрет контекстного меню/скачивания на уровне UX, короткоживущие подписанные URL сегментов, привязка к сессии слушателя.
- B2.2. Динамический водяной знак поверх видео: ФИО + email слушателя полупрозрачно, меняет позицию. Дешёвая и эффективная защита от записи экрана «на продажу».

**ФТ-B3. Прогресс и учёт часов**

- B3.1. События просмотра каждые 10–15 сек → «урок пройден» при пороге (по умолчанию 90%, настраивается).
- B3.2. Опция «антиперемотка» per курс: при первом просмотре нельзя мотать вперёд (типовое требование к нормативным курсам).
- B3.3. Возобновление с места остановки; корректная работа на мобильных.
- B3.4. **Журнал учебной активности**: суммарное время в уроках/тестах по слушателю против часов программы (`program.academic_hours` уже в метаданных версии курса, миграция `0030`). Отчёт «журнал обучения группы» — доказательная база на проверках ГИТ/Минтруда, что 40-часовая программа реально освоена.

**ФТ-B4. Прочий контент**

- B4.1. PDF/презентации — просмотр во встроенном вьювере с фиксацией открытия (тип `'file'` есть).
- B4.2. SCORM: базовая поддержка есть (миграция `0052`, `scorm-zip-guards.ts`) — довести трекинг завершения/оценки в общий прогресс.

**Критерий приёмки:** слушатель на телефоне смотрит курс с видео, прогресс сохраняется, перемотка ограничена (если включено), в карточке группы виден журнал часов; методист загрузил видео сам без техподдержки.

## 5. ЭПИК C — Идентификация слушателей и юридическая значимость **[P0]**

### Зачем это отдельным эпиком

Дистанционное обучение по охране труда юридически держится на двух вещах: (1) УЦ может **доказать, кто именно** учился и проходил проверку знаний, и (2) действия в системе имеют **юридическую силу** (ПЭП/НЭП). У тебя уже реализовано больше половины кирпичей (селфи+паспорт `0050`, pre-exam токены `0044`, прокторинг `0051`, esign с legal log) — эпик собирает их в **настраиваемую политику**, а не разрозненные фичи.

### Требования

**ФТ-C1. Политика идентификации (per tenant, переопределяется per курс/направление)**
Уровни, включаемые галочками в настройках:

- **Уровень 0 — базовый:** уникальный логин + пароль, обязательная смена при первом входе. Всегда включён.
- **Уровень 1 — ПЭП:** при первом входе слушатель подписывает «Соглашение об электронном взаимодействии» (текст редактируется тенантом): фиксируются кто/когда/IP/user-agent/хэш текста. С этого момента клики «Ознакомлен», ответы на тесты и заявления считаются подписанными ПЭП; каждое такое действие пишется в esign legal-log (модуль есть — расширить типы событий).
- **Уровень 2 — подтверждение личности документом:** существующий контур селфи+паспорт с ручной модерацией. Довести UX: очередь модерации у админа (`/admin/identity-verifications` есть), понятные статусы у слушателя, повторная подача при отклонении. Опция «фото непосредственно перед экзаменом» как разовая проверка.
- **Уровень 3 — экзаменационный контроль:** pre-exam токен (одноразовый код) доставляется по email (после ЭПИКа F) и/или СМС (provider seam — ФТ-F2); плюс существующий прокторинг (записи с retention) для выбранных экзаменов.
- C1.5. ЕСИА (Госуслуги): в типах метод `'esia'` уже заложен — оставляем как перспективу [P2], подключение требует соглашений с Минцифры; в ТЗ фиксируем только интерфейсную готовность.

**ФТ-C2. «Личное дело» слушателя**

- C2.1. Сводная вкладка в карточке слушателя: статус идентификации, когда и кем подтверждена, экзаменационные сессии (дата, IP, устройство, длительность), выданные документы, подписанные ПЭП действия.
- C2.2. Экспорт личного дела в PDF — пакет для проверяющих.

**ФТ-C3. Персональные данные и retention**

- C3.1. Снимки селфи/паспорта: retention-cron уже есть (`imagesPurgedAt`) — сделать срок настраиваемым per tenant, решение модератора хранится бессрочно (так уже спроектировано — сохранить).
- C3.2. Согласия: отдельное согласие на обработку ПДн и **отдельное на фото** (юридически чувствительная зона, граничащая с биометрией) — тексты редактируются тенантом, факт согласия — через ПЭП-контур.
- C3.3. Шифрование чувствительных полей (серия/номер паспорта, СНИЛС) на уровне приложения — решение в Открытых вопросах, п. 7.

**ФТ-C4. СНИЛС и реестры**

- C4.1. СНИЛС — ключевой идентификатор для реестра Минтруда: валидация контрольной суммы при вводе, проверка полноты данных **до** формирования выгрузки (`0045`/`0046` уже есть) с понятным списком «у кого чего не хватает».

**Критерий приёмки:** тенант в настройках включает «Уровень 2 + код на экзамен»; слушатель без подтверждённой личности не допускается к итоговому тесту; в личном деле видна полная доказательная цепочка; выгрузка в реестр не собирается, пока у слушателя нет валидного СНИЛС.

## 6. ЭПИК D — Мультитенантность как продукт (аренда) **[P0 изоляция / P1 биллинг]**

**ФТ-D1. Изоляция «до конца» — обязательное условие первого внешнего арендатора**

- D1.1. Tenant-контекст во **всех** контурах: фоновые задачи worker, интеграции, экспорт, realtime (закрывает gap A1). Каждое сообщение в очереди несёт `tenantId`, worker работает строго в скоупе.
- D1.2. Permission-модель на 100% endpoints, включая bulk/export/search (gap A2).
- D1.3. **Тест-суита изоляции** (gap I2): матрица «каждый доменный endpoint × чужой tenant → 403/404», негативные тесты на IDOR по id сущностей, прогон в CI как отдельный `test:isolation`. Опционально вторым слоем — Postgres RLS на критических таблицах (learners, generated_documents, assessment) — практика тебе знакома по prt_ot_doc; включать постепенно, начиная с read-путей.

**ФТ-D2. Жизненный цикл арендатора**

- D2.1. Убрать захардкоженный demo-фоллбек из `TenantService` — единственный источник: БД.
- D2.2. Платформенная админка (только `platform_admin`): список тенантов, создание, статусы `trial → active → suspended → archived`, лимиты, вход «от имени» с обязательной записью в аудит (impersonation).
- D2.3. **Мастер онбординга УЦ**: реквизиты и налоговые данные → номер аккредитации Минтруда и лицензии → логотип/цвета → первая комиссия → стартовые шаблоны документов (из ФТ-A3.6) → демо-курс. Цель: новый УЦ приведён в рабочее состояние за 1 час.

**ФТ-D3. White-label**

- D3.1. Тема тенанта поверх токенов `packages/ui`: логотип, основной/акцентный цвет, название в шапке, подпись в письмах и на публичной странице проверки документов.
- D3.2. Поддомен `<code>.trudskill.<домен>`: резолв тенанта по Host в Next.js middleware; wildcard-сертификат. Собственный домен арендатора — [P2].

**ФТ-D4. Тарифы и лимиты**

- D4.1. Сущности `Plan / Subscription / UsageCounter`: активные слушатели в месяц, хранилище (ГБ — критично из-за видео), число сотрудников УЦ, функциональные флаги (прокторинг, SCORM, API, вебинары).
- D4.2. Экран «Использование» у tenant_admin, предупреждения на 80/95% лимита; мягкая деградация при превышении (нельзя добавить новых слушателей), а не отключение обучения идущих групп.

**ФТ-D5. Биллинг аренды**

- D5.1. В РФ B2B-аренда живёт на «счёт + акт»: в MVP — ручное выставление из платформенной админки (счёт PDF генерирует наш же документный движок — ЭПИК A переиспользуется), фиксация оплат, автоматическое `suspended` через настраиваемый grace-период.
- D5.2. Автоплатежи картой (ЮKassa) — [P1/P2], см. Открытые вопросы, п. 3.
- D5.3. **Не смешивать** с существующим `payments`-модулем (это заказы слушателей/заказчиков на обучение) — биллинг аренды живёт на платформенном уровне отдельными сущностями.

**ФТ-D6. Библиотека контента платформы — главный аргумент продажи аренды**

- D6.1. Каталог готовых курсов платформы (твои курсы по ПП 2464, первой помощи, СИЗ) → копирование или подписка тенанту: новый УЦ получает работающие курсы и шаблоны документов в день подключения.
- D6.2. Механика: «копия» (тенант дальше правит сам) в MVP; «подписка с обновлениями» — [P2].

**ФТ-D7. Tenant health (gap G1)**

- D7.1. Страница платформы: по каждому тенанту — очереди/ошибки задач, использование лимитов, последние выгрузки в реестры, активность. Это твой утренний экран как оператора платформы.

**Критерий приёмки:** создан второй (тестовый внешний) тенант через мастер за ≤1 часа; тест-суита изоляции зелёная; слушатель тенанта А ни при каких манипуляциях с id не видит данных тенанта Б; поддомен работает; счёт за аренду выставлен из админки и после «неоплаты» тенант ушёл в suspended.

## 7. ЭПИК E — Доведение учебного контура **[P1]**

**ФТ-E1. Конструктор курса**

- E1.1. Единый мастер создания: карточка программы (название, шифр, часы, вид обучения, категория слушателей, форма итоговой аттестации — метаданные уже в `0030` и лейблы в `pillar-a-variables.ts`) → модули → уроки (drag&drop порядок) → тесты.
- E1.2. Правила прохождения: последовательное/свободное, «тест доступен после всех материалов модуля», обязательные/необязательные уроки.
- E1.3. Версии курса: группа фиксируется на версии (`course_versions` есть) — изменения курса не трогают идущие группы; понятный UX «создать новую версию».

**ФТ-E2. Тестирование (банк и типы вопросов уже есть — миграции `0040–0043`)**

- E2.1. Довести: рандомная выборка N вопросов из банка, перемешивание вариантов, лимит времени, число попыток, проходной балл — всё настраивается на тесте.
- E2.2. Надёжность прохождения: автосохранение каждого ответа, восстановление попытки после обрыва связи, индикатор соединения, честный таймер на сервере.
- E2.3. Ручная проверка открытых вопросов и возврат попытки на доработку — основа в `0042` есть, довести UX проверяющего.
- E2.4. Импорт вопросов из Excel/GIFT (`/question-import` маршрут есть — довести до устойчивого импорта с отчётом об ошибках).

**ФТ-E3. Комиссия и экзамен → протокол (склейка сквозного сценария)**

- E3.1. Сценарий одной цепочкой: назначить экзамен группе → выбрать комиссию (модуль commissions есть) → слушатели сдают (с идентификацией по политике ЭПИКа C) → результаты фиксируются → **одной кнопкой**: протокол + удостоверения (ЭПИК A) → журнал выдачи → выгрузка в реестр Минтруда/ФРДО (`0045`/`0046`) → номера записей реестра возвращаются в документы.
- E3.2. Валидации до старта: комиссия ≥ 3 человек, у всех членов актуальные данные, у слушателей — СНИЛС (ФТ-C4).

**ФТ-E4. Переобучение (recertification, основа в `0048`)**

- E4.1. Правила периодичности per программа (например, программа Б — 36 мес., первая помощь — 36 мес., работы повышенной опасности — 12 мес.; редактируется тенантом).
- E4.2. Автонапоминания за 60/30/7 дней слушателю, заказчику и менеджеру УЦ (после включения почты). Дашборд «истекающие удостоверения» — в портале заказчика и у менеджера. **Это киллер-фича удержания клиентов и повторных продаж.**

**ФТ-E5. Портал заказчика (`/counterparty-portal` есть — довести)**

- E5.1. Представитель заказчика видит только своих сотрудников: прогресс, документы (скачивание своих PDF), сроки очередного обучения, подача заявок на новые группы (bulk-enrollments есть).

## 12. Нефункциональные требования (дельта к §31 базового ТЗ)

1. Пилотная нагрузка: 300–500 активных слушателей на тенант, 20–50 параллельных экзаменационных сессий, p95 API списков < 300 мс.
2. Видео отдаётся через провайдера/CDN или напрямую из объектного хранилища по подписанным ссылкам — **никогда через backend-процесс**.
3. RTO 4 ч / RPO 24 ч на пилоте; к моменту первых внешних арендаторов — RPO ≤ 1 ч (WAL-архивирование).
4. Генерация пакета документов на группу из 25 слушателей — ≤ 2 минут.
5. Все новые тексты интерфейса — русский язык; локализация не требуется.

| --------------------------------------- | --------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
