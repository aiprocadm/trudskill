# Карта документации и источников правды (SSOT)

Назначение: **зафиксировать, какой файл главный по какому типу информации**, чтобы не размножать противоречивые дубли между `README`, ТЗ и handoff.

## Принцип

У проекта несколько классов истины — это нормально: нельзя смешивать юридически-смысловое ТЗ, реглад инженерной сессии и точку входа для CI в один «мастер-документ».

### Иерархия при расхождении

1. **Согласовано с заказчиком / протокол** (в т.ч. заполненное [приложение Б](../SDOPROF_TZ_FINAL.md#47-приложение-б-фиксация-границ-пилота-must--вне-scope) в `SDOPROF_TZ_FINAL.md`).
2. **Целевые требования к продукту** — [SDOPROF_TZ_FINAL.md](../SDOPROF_TZ_FINAL.md).
3. **Факт реализации в коде** — исходники + тесты; описательно — [LMS_AGENT_HANDOFF.md](../LMS_AGENT_HANDOFF.md) и [TZ_MVP_TRACEABILITY.md](./TZ_MVP_TRACEABILITY.md).
4. **Операционное состояние репозитория для агента** — блок `AI Agent State` в [README.md](../README.md).

## Передача между агентами и фраза «продолжай по ТЗ»

Чтобы разные агенты не расходились в трактовке одной и той же формулировки, используйте **единый порядок чтения** и **двойную фиксацию** итога сессии.

### Что означает «продолжай по ТЗ, смотри документацию»

1. Прочитать блок **AI Agent State** в [README.md](../README.md) — краткая текущая цель и дата.
2. Прочитать [LMS_AGENT_HANDOFF.md](../LMS_AGENT_HANDOFF.md) (§1 дата/ветка, §5 последние пункты, §13 Known Issues, §14/§20 next) — **где остановилась последняя инженерная сессия**: факты, файлы, прогоны тестов.
3. Взять **следующий приоритет требований** из [SDOPROF_TZ_FINAL.md](../SDOPROF_TZ_FINAL.md) (в первую очередь [§41](../SDOPROF_TZ_FINAL.md#41-backlog-укрупнённый) backlog, приёмка [§39](../SDOPROF_TZ_FINAL.md#39-acceptance-criteria)) и сверить с матрицей [TZ_MVP_TRACEABILITY.md](./TZ_MVP_TRACEABILITY.md) (BL → пути в коде).
4. При противоречии текст ↔ код — действовать по **иерархии в начале этого файла** (протокол заказчика выше ТЗ-черновика выше факта кода, если иное не согласовано).

### После каждой инженерной сессии (обязательно согласованно)

- Обновить **README**: `Current Stage`, `Current Goal`, `Last Completed Task`, `Current Task` / `Next Task`, `Last Updated At`.
- Обновить **LMS_AGENT_HANDOFF**: §1 (дата, при возможности коммит), добавить **новый подпункт в §5** или сжатый лог; при необходимости §6–§12; **§20 Final Status** — актуальный срез.

Не держать в README задачи, которые уже закрыты в коде, но не отражены в README: это основной источник ложного «куда продолжать» для следующего агента.

- Полный текст backlog **§41 ТЗ** ведём только в `SDOPROF_TZ_FINAL.md`. В трассировке — **строчные ссылки**, не второй официальный backlog.
- Справочник **новых MVP API** — поддерживать синхронно в [LMS_AGENT_HANDOFF §21](../LMS_AGENT_HANDOFF.md#21-новые-mvp-api-быстрый-справочник) и строкой таблицы **BL-003/007/008** в трассировке.
- Заполнение **приложения Б** (ТЗ §47) → блок **MVP-SCOPE-02** в `TZ_MVP_TRACEABILITY.md`: выполнено по рабочей презумпции **2026-05-03**; при подписи заказчиком обновите оба места одной итерацией.
