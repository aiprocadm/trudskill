# WS1 — Бренд и домен

> **Поправка 2026-06-14:** бренд и домен уже есть — раздел сведён к подтверждению, а не выбору с нуля.

## Цель

Бренд и домен уже есть — подтвердить их и зафиксировать значения `PUBLIC_DOMAIN` и `SMTP_FROM` для запуска сервера (WS2). Выбор названия с нуля не требуется.

---

## Почему это блокирует другое

| Зависит                       | Почему                                                                                                                |
| ----------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| **WS2 — Запуск сервера**      | Значение `PUBLIC_DOMAIN` вписывается в `infra/.env.production` до первого `docker compose up`.                        |
| **WS4 — Бланк удостоверения** | Название и домен бренда печатаются в шапке и подвале документа (удостоверение, сертификат, справка-вызов, программа). |

---

## Чек-лист действий владельца

- [ ] Подтвердить используемый бренд/название.
- [ ] Подтвердить, что домен в собственности и DNS под управлением.
- [ ] Зафиксировать `PUBLIC_DOMAIN` и `SMTP_FROM` для передачи в WS2 (запуск сервера):
  - `PUBLIC_DOMAIN=<ваш-домен>`
  - `SMTP_FROM=noreply@<ваш-домен>` (или выбранный адрес отправителя)

> Если в будущем понадобится ребрендинг — критерии хорошего названия (произносимость, свободный домен, проверка по ФИПС) пригодятся тогда.

---

## Готово когда

Значения бренда/домена/`SMTP_FROM` зафиксированы для конфигурации сервера.
