# WS2 — Запуск сервера и первый вход

**Цель:** платформа живёт по `https://<домен>` с валидным TLS, заведён первый администратор.

---

## Что нужно подготовить заранее

Перед тем как начинать, у вас на руках должно быть следующее:

| Что         | Требования                                                                   |
| ----------- | ---------------------------------------------------------------------------- |
| VPS-сервер  | Ubuntu 22.04+, не менее 4 ГБ RAM, 2 vCPU, 40 ГБ диска                        |
| Домен       | Домен из WS1 (`docs/phase-0/01-brand-domain.md`) с доступом к управлению DNS |
| SMTP-доступ | Хост, порт, логин, пароль и адрес «от кого» для отправки писем               |

> Если домен ещё не выбран или DNS ещё не под вашим управлением, завершите сначала **WS1**.

## Сжатый чек-лист: 10 шагов

Каждый пункт ниже соответствует шагу в [полном руководстве](../../infra/server-setup.md).
Отметьте галочку — и переходите к следующему. Детали, команды и объяснения — там.

- [ ] **Шаг 1. Установить Docker** — установить Docker Engine + плагин Compose на сервере.
      [→ Шаг 1 руководства](../../infra/server-setup.md#step-1--install-docker)

- [ ] **Шаг 2. Настроить firewall (ufw)** — открыть только SSH, 80 и 443; всё остальное закрыть.
      [→ Шаг 2 руководства](../../infra/server-setup.md#step-2--configure-the-firewall-ufw)

- [ ] **Шаг 3. Создать deploy-пользователя + SSH-ключ** — пользователь `deploy` с правами Docker, SSH-ключ для CD.
      [→ Шаг 3 руководства](../../infra/server-setup.md#step-3--create-the-deploy-user)

- [ ] **Шаг 4. Склонировать репозиторий** — клонировать `cdoprof` в `/home/deploy/cdoprof` под пользователем `deploy`.
      [→ Шаг 4 руководства](../../infra/server-setup.md#step-4--clone-the-repository)

- [ ] **Шаг 5. Заполнить `infra/.env.production`** — заполнить все секреты (каждый через `openssl rand -hex 32`), пары паролей (URL ↔ переменная контейнера), домен во всех переменных (`PUBLIC_DOMAIN`, `PUBLIC_BASE_URL`, `CORS_ORIGIN` и др.), SMTP-реквизиты, а также `SCORM_CONTENT_TOKEN_SECRET`.
      [→ Шаг 5 руководства](../../infra/server-setup.md#step-5--configure-the-production-environment)

- [ ] **Шаг 6. Создать DNS A-запись и проверить `dig`** — прописать IP сервера в DNS, дождаться распространения, убедиться что `dig +short <домен>` возвращает нужный IP. Без этого шага Caddy не получит TLS-сертификат.
      [→ Шаг 6 руководства](../../infra/server-setup.md#step-6--dns)

- [ ] **Шаг 7. Первый запуск** — `docker compose -f infra/docker-compose.prod.yml up -d --build`. Миграции БД применяются автоматически при старте бэкенда.
      [→ Шаг 7 руководства](../../infra/server-setup.md#step-7--first-boot)

- [ ] **Шаг 8. Проверить `/api/v1/health/live` + TLS** — убедиться что все контейнеры `healthy`/`running`, `curl https://<домен>/api/v1/health/live` возвращает `{"status":"ok"}`.
      [→ Шаг 8 руководства](../../infra/server-setup.md#step-8--verify)

- [ ] **Шаг 9. Добавить 4 GitHub-секрета для CD** — `DEPLOY_HOST`, `DEPLOY_USER`, `DEPLOY_SSH_KEY`, `DEPLOY_PATH` в `Settings → Secrets and variables → Actions`. После этого каждый push в `main` деплоится автоматически.
      [→ Шаг 9 руководства](../../infra/server-setup.md#step-9--enable-continuous-deployment-github-actions)

- [ ] **Шаг 10. Включить ночной cron бэкапов** — добавить `infra/backup.sh` в crontab пользователя `deploy` (дамп PostgreSQL + снапшот MinIO каждую ночь, хранение 14 дней).
      [→ Шаг 10 руководства](../../infra/server-setup.md#step-10--enable-automated-backups)

## Первый администратор

После того как сервер поднят, выполните процедуру из **[bootstrap-admin.md](../../infra/bootstrap-admin.md)**:

1. Обновите email тестового пользователя `u_tenant_admin` на ваш реальный адрес.
2. Заблокируйте остальных demo-пользователей (хорошая практика безопасности).
3. Войдите через magic-link: откройте `https://<домен>/login`, введите email, перейдите по ссылке из письма.

> **Важно:** дефолтный seed-пароль (`Password123!`) автоматически нейтрализуется при первом запуске в production-режиме — механизм `SeedCredentialHygiene` меняет хэш на недействительный у всех seed-аккаунтов. Поэтому войти с паролем не получится даже если попробовать. Первый вход — только по magic-link согласно процедуре в `bootstrap-admin.md`.

## Чем помогает агент

Если что-то пошло не так, агент может:

- провести по шагам интерактивно, объяснить каждую команду простым языком;
- помочь правильно заполнить `infra/.env.production` (какие секреты генерировать, какие пары значений должны совпадать);
- продиагностировать ошибки health-чека, TLS или Caddy — попросите скинуть вывод `docker compose logs caddy` или `docker compose ps`;
- помочь разобраться, почему не приходит письмо с magic-link (SMTP-настройки, `NOTIFICATIONS_EMAIL_ENABLED`).

## Готово когда

- [ ] `curl https://<домен>/api/v1/health/live` возвращает `{"status":"ok"}`.
- [ ] Вход первого администратора по magic-link работает, открывается кабинет администратора.
- [ ] Деплой по push в ветку `main` проходит зелёным в GitHub Actions (CD).
