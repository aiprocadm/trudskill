## 2. Утверждённый дизайн — pluggable сканер (Approach A)

### 2.1 Абстракция

Новый каталог `apps/backend/src/infrastructure/antivirus/`:

```ts
// antivirus.scanner.ts
export type AntivirusVerdict = 'clean' | 'infected' | 'error';

export interface ScanResult {
  verdict: AntivirusVerdict;
  detail?: string; // имя сигнатуры при infected, текст ошибки при error
}

export interface AntivirusScanner {
  scan(params: { key: string }): Promise<ScanResult>;
}

export const ANTIVIRUS_SCANNER = Symbol('ANTIVIRUS_SCANNER');
```

- **`NoopAntivirusScanner`** (dev/test/in-memory по умолчанию): всегда `{ verdict: 'clean' }`. Никакой инфраструктуры не требует — локальная разработка и весь in-memory MVP работают как раньше.
- **`ClamAvAntivirusScanner`** (prod): тянет объект из S3/MinIO по `key` и стримит в `clamd` (INSTREAM по TCP `CLAMAV_HOST:CLAMAV_PORT`). При недоступности clamd → `{ verdict: 'error' }` (fail-closed на уровне gate, см. §5).

### 2.2 Выбор реализации по env (зеркало `MVP_PERSISTENCE_BACKEND`)

Factory-provider на токене `ANTIVIRUS_SCANNER` в `FilesModule`:

```ts
{
  provide: ANTIVIRUS_SCANNER,
  inject: [S3StorageClient],
  useFactory: (storage: S3StorageClient) =>
    backendEnv.ANTIVIRUS_ENABLED
      ? new ClamAvAntivirusScanner(storage, backendEnv.CLAMAV_HOST, backendEnv.CLAMAV_PORT)
      : new NoopAntivirusScanner()
}
```

Новые env (в `env.ts` + `.env.example`): `ANTIVIRUS_ENABLED` (boolean, default `false`), `CLAMAV_HOST`, `CLAMAV_PORT` (default `3310`).

### 2.3 Почему так

- **Тестируемость безопасности отдельно от инфраструктуры.** Поведение gate («не отдавать не-clean файл») на 100% покрывается unit-тестами с fake-сканером — без поднятого ClamAv. Подключение реального движка — это смена флага + ops-задача (поднять clamd-контейнер), а не переписывание логики.
- **Ноль регрессий в dev/тестах.** `NoopAntivirusScanner` помечает `clean` → существующий поток Plan C (upload → attach → download) продолжает работать без изменений в окружениях, где AV выключен.
- **Согласованность с кодовой базой.** Тот же приём «интерфейс + выбор по env», что и для персистентности.

## 3. Поток скана и enforcement

### 3.1 Жёсткая гарантия (must-have): gate при скачивании

Оба пути скачивания возвращают presigned URL **только** если `antivirus_status === 'clean'`:

- `FilesService.createDownloadUrl(tenantId, fileId)` — добавляет проверку статуса (defense in depth: даже если вызовут в обход MvpService).
- `MvpService.getSubmissionFileUrl(...)` — наследует гарантию через делегирование.

Логика gate при чтении файла:

| Текущий `antivirus_status` | Поведение download                                                                  |
| -------------------------- | ----------------------------------------------------------------------------------- |
| `clean`                    | вернуть presigned GET URL                                                           |
| `infected`                 | отказ `423` `{ code: 'file_infected' }` — никогда не отдаём                         |
| `error`                    | отказ `409` `{ code: 'file_scan_failed' }` — нужен повторный скан                   |
| `pending`                  | **ленивый fallback:** прогнать `scanFile` сейчас, затем повторить разбор по статусу |

Ленивый fallback в ветке `pending` — страховка: даже если проактивный скан при submit не отработал (медленный clamd, фоновый сбой), файл **никогда** не утечёт нескан­ированным. С `NoopAntivirusScanner` это мгновенно (`clean`).

### 3.2 Проактивный скан (best-effort): при submit

`submitAssignmentSubmission` после перевода в `submitted`, если у сдачи есть `fileId`, инициирует `filesService.scanFile(tenantId, fileId)`. Цель — к моменту проверки ревьюером файл уже `clean`/`infected`, без ожидания на read-пути.

> **Замечание по async.** `getSubmissionFileUrl`/`createDownloadUrl` уже `async` — жёсткий gate (§3.1) самодостаточен и не зависит от того, синхронен ли submit. Проактивный триггер — оптимизация; план выберет наименее инвазивный способ его подключить (await в submit, либо тот же fire-and-forget-паттерн, что `ensureMaterialLink` на [mvp.service.ts:1008](../../../apps/backend/src/modules/mvp/mvp.service.ts), при условии что ленивый fallback закрывает гонку).

## 4. Статусы и переходы

```
pending ──scanFile──▶ clean      (отдаётся)
        ──scanFile──▶ infected   (навсегда заблокирован; пере-скан не реабилитирует)
        ──scanFile──▶ error      (можно пере-сканировать: error ──scanFile──▶ clean/infected)
```

`infected` — терминальный для скачивания. `error` — повторяемый (clamd мог быть временно недоступен). Значения уже разрешены CHECK-констрейнтом миграции 0002 — **новой миграции нет**.

## 5. Обработка ошибок и позиция безопасности

- **Коды отказа gate:** `file_infected` (423 Locked), `file_scan_failed` (409 Conflict для `error`), существующие `no_file` / `file_not_found` сохраняются. Обёртка — стандартный `HttpExceptionEnvelopeFilter`.
- **Fail-closed при включённом AV:** если `ANTIVIRUS_ENABLED=true`, а clamd недоступен → `scanFile` пишет `error` → download отказывает (а не «на всякий случай отдаёт»). Безопасность важнее доступности для регулируемого домена.
- **Осознанная позиция при выключенном AV (`ANTIVIRUS_ENABLED=false`, дефолт):** `NoopAntivirusScanner` помечает `clean`, gate пропускает. Это **сознательный** выбор для dev/пилота без поднятого clamd — механизм на месте, реальная защита включается флагом. Задокументировано как известное ограничение (см. README §13 Known Issues при закрытии).

## 6. Frontend — минимальное отражение статуса

Enforcement — на бэкенде; фронт лишь корректно отражает:

- **Read-DTO сдачи** (`getAssignmentSubmission` / элемент очереди ревьюера для submissions) экспонирует `antivirusStatus` файла (когда `fileId` задан).
- **Reviewer UI** ([reviewer-actions-screen.tsx](../../../apps/frontend/src/features/reviewer-actions/reviewer-actions-screen.tsx)): кнопка «Скачать файл» показывается только при `clean`; иначе статус-текст («файл на проверке» / «файл заблокирован (заражён)» / «ошибка проверки»).
- **Learner UI** ([practical-submissions](../../../apps/frontend/src/features/practical-submissions/)): после прикрепления файла показывать его AV-статус.
- Если фронт всё же дёрнет download для не-`clean` файла — корректно показать ошибку из envelope, а не открыть битый URL.

## 7. Тестирование

| Уровень                 | Что проверяем                                                                                                                                                    |
| ----------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Scanner unit            | `NoopAntivirusScanner` → `clean`; `ClamAvAntivirusScanner` с замоканным сокетом/стримом → разбор `clean/infected/error`                                          |
| `FilesService` unit     | `scanFile` пишет статус+`checked_at`; `createDownloadUrl` отдаёт при `clean`, отказывает при `infected`/`error`, делает ленивый скан при `pending` (fake-сканер) |
| `MvpService` unit       | `getSubmissionFileUrl` наследует gate; submit инициирует скан                                                                                                    |
| HTTP integration        | `plan-c.http.integration.test.ts` расширить: `file-url` отдаёт при `clean`, отказывает `file_infected`/`file_scan_failed`; permission-границы без изменений      |
| Frontend contract + e2e | API-контракт `file-url` (envelope + коды); reviewer-actions прячет «Скачать» для не-`clean`; e2e роутинг/nav без регрессий                                       |

Все тесты — изолированными файлами с `--no-file-parallelism` (Cyrillic-path gotcha, CLAUDE.md).

## 8. Вне объёма (осознанно отложено)

- **Partial credit для multiple-choice.** Изменение политики оценивания; в регулируемых аттестациях экзамены обычно строго pass/fail — нужно отдельное решение владельца о схеме (и opt-in per test). Отдельная итерация.
- **Essay max-score hint в reviewer UI.** Показ потолка балла рядом с полем ввода — мелкий UX-полиш, не блокирует безопасность. Backlog.
- **Реальное развёртывание ClamAv.** Поднять clamd-контейнер в Yandex Cloud, выставить `ANTIVIRUS_ENABLED=true`, настроить ресурсы/обновление баз сигнатур — ops-задача (Phase 0/11), не код.
- **MinIO CORS / bucket-policy** для presigned PUT — уже отмечено как операционная задача в Plan C, здесь не меняется.
