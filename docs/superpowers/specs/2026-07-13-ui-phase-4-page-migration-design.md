# UI redesign — Фаза 4: миграция экранов на каркасы (page migration)

- **Дата:** 2026-07-13
- **Статус:** дизайн согласован (brainstorming), исполнение по пачкам после merge Фазы 3
- **Зависимость:** каркасы `@trudskill/ui` из Фазы 3 (§5.164, PR #302) — **должны быть в `main`** до старта первой пачки.
- **Ветка(и):** по одной на пачку, `feat/<YYYY-MM-DD>-ui-phase-4-<domain>`, от актуального `main`.
- **Предшественники:** Фаза 1 (дизайн-система), Фаза 2 (навигация), Фаза 3 (эталонные шаблоны + 4 пилота).

---

## 1. Контекст и цель

Фаза 3 дала 6 переиспользуемых каркасов (`AsyncSection`, `StatGrid`, `DetailLayout`, `ListPage`, `Form`/`FormSection`/`FormActions`, `SelectField`) и доказала их на 4 пилотах. Фаза 4 — **массовая механическая миграция** оставшихся экранов (~74 из 78, минус 4 пилота) на эти каркасы, пачками, без изменения поведения.

**Цель:** привести все экраны 4 архетипов (dashboard/список/карточка/форма) к единой композиции, попутно вычистив накопленные расхождения (сырые `<input>`, ручные KPI/KV/state-каскады, `window.confirm`, инлайновые `style={{}}`), **не меняя ни поведения, ни данных, ни прав, ни URL**.

## 2. Объём

**Входит:** миграция экранов на каркасы Фазы 3 + попутные нормализации (перечислены в §4, п.3), по доменным пачкам (§3).

**Не входит (осознанно):**

- **Слияние дублирующих экранов** и рискованные объединения (`/learners`↔`/admin/learners`, `/counterparties`↔`/admin/clients`, `/reports`↔`/admin/analytics`, `/student/dashboard`→`/learner`, esign-вкладки, `/question-import`, `/sync-logs`) — это **Фаза 5** (по per-item решению владельца).
- Тёмная тема / адаптив / полный a11y-проход, включая accname-фикс общего `.ui-field` — **Фаза 6**.
- Архетипы «плеер»/«мастер»/«прочее» (тест-плеер, прокторинг-плеер, bulk-enrollments, авторизация/ошибки) — не шаблонизируются (спец-поверхности).
- Любые изменения URL, `routeMeta`/RBAC, бэкенда, миграций, прав.

**Уже сделано в Фазе 3 (пилоты, не переделывать):** `learners-list-screen`, `client-detail-screen`, `academy/requisites`, `analytics/screens.tsx` (KPI+состояния).

## 3. Порядок пачек (каждая = отдельный PR, от актуального `main`)

Порядок по рычагу и связности. Точный список экранов в каждой пачке уточняется в её just-in-time плане (см. §6).

1. **Монолит `mvp/screens.tsx`** — самый рычажный файл: `CoursesPageScreen` (список), `CourseDetailsScreen` (карточка), `CourseCreateScreen` (форма), `AdminCockpitScreen`/`StudentDashboardScreen`/`TeacherGradingCenterScreen` (dashboard через `RoleWidgetGrid`), `DirectionsPageScreen`, `CommissionDetailsScreen`, `LearnerCourseDetailsScreen` и пр. Разбить мегафайл при миграции допустимо (по экрану/архетипу), если это снижает связность.
2. **Ученики / компании / группы** — доделать после пилотов: детали ученика, список компаний, группы (list+detail+form).
3. **Assessment** — тесты / банки вопросов / задания (admin+learner), list+detail+form.
4. **Документы / esign / госвыгрузки** — `documents`, `esign/*`, `gov-export`, `exports`, `sync-logs`, `registry`.
5. **Оплаты / переаттестация / идентификация / прокторинг** — `admin/orders`, `admin/payments/settings`, `admin/recertification`, `identity`, `proctoring` (списки/настройки).
6. **Аналитика / отчёты / аудит / оперпанель** — `report-builder`, `audit`, `workspace`, `app/page.tsx` (root dashboard) — analytics-KPI уже пилот.
7. **Коммуникации** — `chat`, `notifications`, `notification-settings`, `webinars` (admin+learner), `telephony`, `mailings`.
8. **Остаток** — `academy` (кроме requisites), `settings`, `scorm`, `materials`, `learning/calendar`, `counterparty-portal`, `integrations`, `directions` (если не в п.1), `users`.

## 4. Рецепт миграции одного экрана

1. **Определить архетип** (dashboard/список/карточка/форма) и подставить каркас: `ListPage` / `DetailLayout` / `Form`+`FormField`/`SelectField` / `StatGrid`; состояния — через `AsyncSection`.
2. **Внедрить простаивающие примитивы** там, где экран рисует руками: `StatCard` (KPI), `KeyValueList` (пары ключ-значение вместо `dl.ui-data-list`/`<dl>`), `Button` (вместо сырых `<button className="ui-button…">`).
3. **Попутные нормализации:** `window.confirm`/`confirm()` → `ConfirmDialog`; инлайновые `style={{}}` с цветами/отступами → токены/классы (но не задевать общие классы, используемые другими экранами — как в пилоте `academy/requisites`); сырые `<input>`/`<label>` → `FormField`/`TextareaField`/`SelectField`.
4. **Сохранить точно (инвариант «zero behavior change»):** все обработчики, валидации, условия видимости/прав (`hasPermission`/`PermissionWrapper`), хуки данных (React-Query и `useState` — оба контракта каркасы принимают), URL/маршруты, drawer'ы/модалки, тексты.
5. **Контракт данных:** адаптер к форме хука — на стороне экрана (`ListPage` ждёт `{items,total,isLoading,error,refetch}`; строковую ошибку оборачивать `new Error(...)` для `AsyncSection`, как в analytics-пилоте).

## 5. Definition of Done (на пачку/PR)

- Все экраны домена переведены на каркасы; ручных `state`-каскадов/KPI/KV в затронутых файлах не осталось.
- **Zero behavior change:** поведение/данные/права/URL идентичны (проверяется adversarial-ревью с фокусом на регрессии, как в Фазе 3).
- Гейты зелёные: `pnpm --filter @trudskill/frontend exec vitest run`, `pnpm typecheck`, `npx eslint <затронутые> --max-warnings=0`; страж токен-дисциплины (если трогали CSS).
- Существующие e2e (permission/routing, import-smoke) для затронутых экранов зелёные.
- README §2 + handoff §5.NN обновлены; без миграций/новых прав/бэкенда.
- (Если поднят стек) визуальная проверка ключевых экранов в браузере.

## 6. Модель исполнения

- **Just-in-time план на каждую пачку** перед её выполнением (не планировать все 78 экранов разом — YAGNI). План пачки: список экранов, их архетипы, per-screen рецепт, критерии.
- Исполнение — **subagent-driven** (как Фаза 3): реализация субагентами по TDD/рецепту → полные гейты → **adversarial-ревью диффа с фокусом на потерю поведения** (находки перепроверяются скептиками) → доки → PR по домену.
- Каждая пачка — своя ветка от актуального `main` (после merge предыдущей пачки), свой PR. Владелец ревьюит/мёрджит между пачками (phase-by-phase контроль).

## 7. Риски и как их снимаем

- **Главный риск — тихая потеря поведения** при переписывании композиции (пропущенный обработчик/валидация/условие прав). Снятие: явный инвариант §4 п.4 + adversarial-ревью каждой пачки, целящееся именно в регрессии (сравнение с оригиналом через `git show BASE:<file>`).
- **Мегафайл `mvp/screens.tsx`** — большой и связный. Снятие: допускается разбиение по экранам/архетипам при миграции; пачка №1 может быть разбита на под-PR, если дифф слишком велик для ревью.
- **Общие CSS-классы** (напр. `.ui-form` в `foundation.ts`, используемый 5+ формами): не менять глобально ради одного экрана (прецедент пилота `academy/requisites` — инлайн-ширина сохранена).
