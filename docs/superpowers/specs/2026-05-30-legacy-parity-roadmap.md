## 1. Назначение и метод

Владелец прислал **8 инструкций действующей платформы СДО ПРОФ** (инкумбент, cdoprof.ru) — прокторинг, SCORM-тренажёры, НЭП, общий FAQ и базовую инструкцию. Эти документы — фактический **эталон функционала**, который новая платформа CDOProf должна закрыть, чтобы УЦ мог отключить старую систему без потери возможностей и без срыва обязательной отчётности.

Метод: каждая фича из инструкций сопоставлена с (а) реальным кодом монорепозитория (модули backend/frontend, миграции), (б) существующим ТЗ (§13–§23, backlog BL-001…012). Каждой строке присвоен статус, привязка к ТЗ, приоритетный «тир», предварительная оценка трудозатрат и перечень **внешних входов** (то, что должен предоставить владелец/заказчик, прежде чем фичу можно делать).

**Легенда статуса:** ✅ есть · 🟡 частично (есть кирпичики/каркас) · ❌ нет.
**Легенда трудозатрат (предварительно, в терминах ваших итераций-«планов» ≈ 10–13 задач):** S ≈ ≤1 плана · M ≈ 1 план · L ≈ 2–3 плана · XL ≈ несколько фаз.

## 5. Открытые решения (нужен вход владельца/заказчика)

Эти развилки **гейтят** соответствующие фичи — без ответа их нельзя корректно специфицировать:

1. **Прокторинг (#7):** интеграция с **ProctorEdu** (как у инкумбента, быстрее, есть готовый протокол) или собственная реализация (дороже, полный контроль)? Это решение определяет всю Phase 4.
2. **НЭП (#8):** требуется зафиксировать **юридическую модель** (регламент НЭП/КЭП, хранение, доказательная цепочка — ТЗ §44.3, BL-011) до реализации крипто-подписания.
3. **SCORM/тренажёры (#11):** реактивировать ли SCORM-scope (сейчас Should/Post-MVP по §2.5)? Нужны ли тренажёры LContent конкретно, или достаточно собственных практических заданий?
4. **Регуляторные шаблоны (#1, #2, #3):** предоставить **актуальные официальные шаблоны** — Excel ФИС ФРДО, XML-схему ЕИСОТ, XML ЛКОТ, и **ключ API ЛКОТ Минтруда** (у действующего УЦ они есть).
5. **Медиа-стек вебинаров (#10):** Jitsi (self-host, бесплатно), LiveKit, или сторонний (Zoom/иной) API?
6. **Внешние сервисы Тира 4:** нужны ли DaData (ИНН), Мегафон ВАТС, SendPulse — и есть ли по ним учётки/бюджеты?

## 6. Осознанно вне ближайшего scope (фиксируем, чтобы снять ожидания)

- **SSO / корпоративный IdP** (BL-009) — вне пилота без согласованного протокола.
- **Нейро-генераторы** (#16, BL-012) — post-MVP/Could; ценны для производства контента, но не блокируют паритет.
- **Конструктор сайта, телефония, полноценный CRM** (#17, #21, #23) — коммерческий контур, после паритета по обучению.
- **Юридически значимое подписание** (#8 в полном объёме) — только после фиксации юр. модели (§19, BL-011).

Это согласуется с §47.3–47.4 ТЗ; данная дорожная карта **не меняет** эти решения, а делает их явными в одном месте.

## 7. Следующий шаг

По выбору владельца — **детальная спецификация Волны 1** («Учебно-экзаменационное соответствие»): модульность с последовательным прохождением + время на изучение материала + аутентификация перед экзаменом (Приказ №816). Это отдельный шаг (brainstorming → spec → plan), запускается после ревью этой дорожной карты.

Остальные волны specифицируются по мере приоритизации; развилки §5 разрешаются до старта соответствующих волн.
