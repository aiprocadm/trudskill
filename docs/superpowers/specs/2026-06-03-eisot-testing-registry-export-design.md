> **⚠️ Ключевое отличие от выгрузок A и B.** ОТ-реестр (B) строит строки из **завершённых зачислений + сданного протокола**; ФРДО (A) — из **выданных документов**. ЕИСОТ «лица на тестирование» (C) — это **ростер «кого предстоит протестировать», формируемый ДО сдачи**: список слушателей по фильтру (группа / период / клиент), без привязки к результату экзамена или к выданному документу (их ещё нет). Поэтому источник — **зачисления → слушатели** (`MvpService`-геттеры), гранулярность — **одна строка на слушателя** (дедуп), обратной загрузки нет. Это самая лёгкая из трёх выгрузок: **новой миграции не требуется** — все поля (`snils`/`position`/`dateOfBirth` слушателя, `inn` работодателя) уже в `main`. Остальная форма (durable MVP-модуль, preflight, golden-file `.xlsx`, provisional-маппинг, UI-секция в `gov-export`, права `regulatory.export.*`) переиспользуется из ФРДО почти 1:1.

## 1. Назначение и зафиксированные решения

Учебный центр направляет слушателей на проверку знаний требований охраны труда (тестирование) и обязан передавать сведения о направляемых лицах в систему Минтруда (ЛКОТ / ЕИСОТ). Сейчас это делает прежняя платформа; чтобы УЦ мог уйти с неё, CDOProf должен **сам формировать корректный файл-ростер** направляемых на тестирование для загрузки в ЛКОТ.

Это третья реализуемая под-цель Волны 2 (B = реестр обученных по ОТ — слита, #222/#223; A = ФИС ФРДО — слита, #225; **C = ЕИСОТ «лица на тестирование» — этот дизайн**). Под-цели B и A задали прецедент, инфраструктуру и провизорный паттерн (точки-swap'а + баннер).

**Решения, принятые в брейншторме (зафиксированы, одобрены владельцем 2026-06-03):**

1. **Архитектура — durable MVP-модуль, как у ФРДО/ОТ.** Новый модуль `apps/backend/src/modules/mvp/eisot-testing-registry/` зеркалит `frdo-registry/`. Пустой `EisotAdapter` в `integrations/` **не наполняется** — оркестратор `integrations` не персистентен и пишет один summary-item, что не даёт пер-строчных preflight-ошибок (то же отклонение D1, по которому ОТ и ФРДО ушли от адаптера). Адаптерный шов остаётся нетронутым под будущую live-API/ЕСИА-отправку (Phase 4).
2. **Источник данных — все слушатели по фильтру (ростер), не экзамены и не документы.** Берём зачисления (`enrollments`) по фильтру (группа / период `enrolledAt` / клиент) → их слушателей; **без привязки к статусу экзамена** (его ещё нет). Работодатель (наименование/ИНН) — из клиента (counterparty) группы.
3. **Гранулярность — одна строка на слушателя.** Дедуп по слушателю в рамках батча. Если слушатель попал в несколько отобранных групп — берётся его зачисление в первой отобранной группе (работодатель/программа — оттуда); это provisional-правило, уточняется при сверке с эталоном.
4. **Формат — Excel (`.xlsx`) основной, XML отложен.** Штатная ручная загрузка ростера — Excel-шаблон, проще для глазной сверки. XML по схеме ЛКОТ и прямой веб-сервис требуют ЕСИА/УКЭП и статуса ИС — это Phase 4. XML добавляется позже одной правкой (как у ОТ, §16 её дизайна), из того же ряда-источника. _(Дефолт, одобрен владельцем; roadmap помечал ЛКОТ как «XML», но ручной канон — Excel.)_
5. **Обратной загрузки нет.** Ростер направляемых на тестирование — это входной список; ЛКОТ не присваивает по нему рег. номера записей. Раунд-трип (как в ОТ) не требуется. _(Дефолт, одобрен владельцем.)_
6. **Provisional-формат — 1 точка-swap'а + баннер.** Официальный шаблон/схема ЛКОТ есть у УЦ, но не предоставлены. До сверки маппинг поле→колонка помечен `PROVISIONAL`; подмена на эталон = правка одной константы (`COLUMNS` в writer'е), без изменения логики сбора/preflight.

## 2. Что уже готово

**Источник — учётные данные слушателей/зачислений** (`mvp.types.ts`, `MvpService`):

- Слушатель: `firstName`/`lastName`/`middleName?` (ФИО), `snils?` (валидатор ПФР, `learners-bulk-import.service.ts`), `position?` (должность), **`dateOfBirth?`** (добавлено в ФРДО, PR #225 — уже в `main`).
- Зачисление: `Enrollment { groupId, learnerId, status, enrolledAt }`, `EnrollmentStatus = 'pending'|'active'|'suspended'|'completed'|'cancelled'`. Геттеры/листеры `MvpService` (`listEnrollments`/`getLearner`/`getGroup`) — фильтрация по группе/клиенту/периоду уже используется ОТ-реестром.
- Работодатель: клиент группы (`group.counterpartyId` → `counterparty.inn`, 10/12 цифр, DB CHECK — Phase 2 C).
- Программа: курс группы (`groupCourses` → `course.title`) — provisional-источник наименования программы/категории проверки знаний.

**Инфраструктура регуляторных выгрузок** (прецедент ФРДО/ОТ): durable MVP-модули `frdo-registry/` и `ot-registry/` (service/writer/preflight/rows/dto/controller/tests), права `regulatory.export.read/write` (миграция `0045`), UI-секции в `app/gov-export/page.tsx`, фронт-фича `features/gov-export/` (api/hooks/types + contract-test + e2e-смоук) — зеркалятся.

**Заглушка, которая ВЫТЕСНЯЕТСЯ (не наполняется):** `EisotAdapter` (integrations) и общий мастер `gov-export` (provider=`eisot`) — остаются как есть; C реализуется durable-модулем и отдельной UI-секцией рядом с ФРДО/ОТ.

**Пробелов в данных нет** — в отличие от ФРДО (требовала `date_of_birth`), C полностью покрыта существующей схемой.

## 3. Поля ростера и источники

Одна строка = один слушатель, направляемый на тестирование:

| Поле ростера                                       | Обяз. | Формат           | Источник в CDOProf                              |
| -------------------------------------------------- | ----- | ---------------- | ----------------------------------------------- |
| Фамилия / Имя / Отчество                           | ✔     | текст            | слушатель (`lastName`/`firstName`/`middleName`) |
| СНИЛС                                              | ✔/○   | `XXX-XXX-XXX YY` | `learner.snils` (валидатор ПФР)                 |
| Дата рождения                                      | ○     | `ДД.ММ.ГГГГ`     | `learner.dateOfBirth`                           |
| Должность                                          | ○     | текст            | `learner.position`                              |
| Работодатель (наименование)                        | ✔     | текст            | клиент группы (`counterparty.name`)             |
| ИНН работодателя                                   | ✔/○   | 10/12 цифр       | `counterparty.inn`                              |
| Наименование программы / категория проверки знаний | ○     | текст            | курс группы (`course.title`; provisional)       |
| Дата направления                                   | ○     | `ДД.ММ.ГГГГ`     | `enrollment.enrolledAt`                         |

> Точный набор/порядок колонок и кодировка значений пиннятся по официальному шаблону/схеме ЛКОТ (§14). Маппинг поле→колонка изолирован в одной константе (`COLUMNS`), поэтому уточнение шаблона — drop-in.

## 4. Архитектура

**Durable MVP-модуль `eisot-testing-registry/`** (зеркало `frdo-registry/`). Вся ЕИСОТ-специфика:

- `eisot-testing-registry.service.ts` (`Scope.REQUEST`) — собирает зачисления по фильтру через `MvpService`-геттеры, джойнит слушателя/группу/клиента/курс, дедуплицирует по слушателю, строит строки, валидирует (preflight), генерирует `.xlsx`, персистит батч/записи (как ФРДО).
- `eisot-testing-rows.ts` — чистая функция `buildEisotTestingRows(bundles)`: одна строка на слушателя; форматирование дат (`ДД.ММ.ГГГГ`).
- `eisot-testing-preflight.ts` — чистые валидаторы (§9): СНИЛС (ПФР, при наличии), обязательные поля (ФИО, работодатель), ИНН (при наличии).
- `eisot-testing-xlsx.writer.ts` — ExcelJS-генератор с `COLUMNS` (единственная точка-swap маппинга поле→колонка, помечена `PROVISIONAL`).
- `eisot-testing-registry.controller.ts` — эндпоинты (§7) под `regulatory.export.read/write`.
- DTO `CreateEisotTestingExportDto` (фильтры: период `from`/`to`, группа `groupId`, клиент `clientId`).
- Типы (`EisotTestingRow`, `EisotTestingRowError`, `EisotTestingBatch`, `EisotTestingRecord`, `EisotTestingExportOutcome`) в `mvp.types.ts`.

Батч/записи регистрируются в request-scoped MVP-state-коллекциях `eisotTestingBatches`/`eisotTestingRecords` — **не забыть зарегистрировать их в `mvp-collections.ts`** (иначе теряются между HTTP-запросами; этот нюанс ловил Pillar A Plan A Task 10).

## 5. Альтернативы (отвергнуты)

- **Наполнить `EisotAdapter`** в `integrations/` — отвергнуто: оркестратор не персистентен, 1 summary-item, нет пер-строчных preflight-ошибок (то же отклонение D1, что у ОТ/ФРДО).
- **Источник из экзаменов/назначенных тестов** (assessment-модуль) — отвергнуто владельцем: завязывает ростер на то, что тесты уже назначены каждому; «лица на тестирование» = входной список, экзамена ещё нет.
- **Источник из выданных документов** (как ФРДО) — неприменимо: документы появляются после успешной проверки, а ростер нужен до неё.
- **Расширить `ot-registry/` ещё одним `exportType`** — отвергнуто: смешивает «обученных» (выход) и «направляемых на тестирование» (вход) с разной логикой отбора в одном модуле; хуже для понимания и тестирования.

## 6. Изменения в данных — отсутствуют

**Миграции не требуется.** Все поля уже в `main`:

- `learner.snils` / `learner.position` (Pillar A Plan C), `learner.dateOfBirth` (ФРДО, #225);
- `counterparty.inn` (Phase 2 C);
- права `regulatory.export.read/write` (миграция `0045`) — **переиспользуются** (та же `/gov-export`-поверхность и тот же контур доступа к ПДн-выгрузкам).

Наименование программы/категория проверки знаний берётся provisional из `course.title`. Если позже понадобится **строгий классификатор категорий** — переиспользуем уже засеянный `lookup.ot_training_programs` (сид `0045`) или добавим следующую свободную миграцию (после `0046`); это **осознанно отложено** (§15), в V1 не требуется.

## 7. Поток выгрузки (outbound)

1. `POST /eisot-testing-registry/exports` с `{ from?, to?, groupId?, clientId? }`.
2. Сервис собирает зачисления по фильтру (период по `enrolledAt`, группа, клиент; **исключает `cancelled`** — отозванных не направляют на тестирование), джойнит слушателя (ФИО/СНИЛС/дата рождения/должность), клиента-работодателя (наименование/ИНН), курс группы (наименование программы), **дедуплицирует по слушателю**.
3. **Preflight-валидация** (§9), принцип частичного успеха: валидные строки → файл; невалидные → пер-строчные ошибки на записи; батч → `generated` / `partial` / `failed`.
4. `eisot-testing-xlsx.writer` формирует `.xlsx` по provisional-шаблону → сохраняет в `storage.files`; `batch.fileId` проставлен.
5. Владелец скачивает файл из секции `gov-export`, при необходимости подписывает и загружает в ЛКОТ.

Синхронно для разумных партий через текущий жизненный цикл (асинхронный воркер — если потребуется по объёму; решает план).

## 8. Обратная загрузка — отсутствует

ЛКОТ не присваивает и не возвращает номера по входному ростеру направляемых на тестирование. Раунд-трип не реализуется (в отличие от ОТ-реестра обученных). Если по факту ЛКОТ отдаёт файл-квитанцию о приёме — парсер можно добавить позже (§15), как у ФРДО.

## 9. Preflight-валидация

- **Жёстко (строка в ошибки):** есть ФИО (фамилия+имя); есть наименование работодателя.
- **Мягко (строка идёт в файл, пустая ячейка, без ошибки):** отсутствует СНИЛС; отсутствует дата рождения; отсутствует должность; отсутствует ИНН; отсутствует наименование программы. (Provisional — жёсткость поднимется при сверке с эталоном ЛКОТ.)
- **СНИЛС** при наличии — алгоритм ПФР (переиспользовать `isValidSnilsChecksum`/`normalizeSnils` из `learners-bulk-import.service.ts`).
- **ИНН** при наличии — 10/12 цифр.

## 11. Права и безопасность

- Переиспользуются `regulatory.export.read` (чтение/скачивание) и `regulatory.export.write` (формирование) — миграция `0045`. **Новых прав нет.**
- **Файл содержит ПДн** (ФИО, СНИЛС, дата рождения, должность, работодатель) — скачивание под `regulatory.export.read`; учитывать антивирус-гейт скачивания (V1.1) и аудит на формирование/скачивание (`regulatory.eisot_testing_exported`, `regulatory.eisot_testing_downloaded` — по конвенции `regulatory.*` из ФРДО/ОТ).

## 12. Тестирование

Трио + специфика (зеркало ФРДО):

- **Rows unit** — `buildEisotTestingRows`: одна строка на слушателя, формат `ДД.ММ.ГГГГ`, пустые ячейки для отсутствующих опциональных полей.
- **Preflight unit** — табличные кейсы (нет ФИО; нет работодателя; битый СНИЛС → ошибка; пустой СНИЛС/ИНН → ок).
- **Service unit** — джойн зачисление→слушатель/клиент/курс, дедуп по слушателю, исключение `cancelled`, частичный успех, персистенция батча/записей + файл.
- **Golden-file `.xlsx`** — фиксирует точный набор/порядок колонок и формат значений (читает структуру обратно).
- **DTO validation** — форма `CreateEisotTestingExportDto`.
- **HTTP integration** — границы прав (`regulatory.export.read/write`) на стаб-контроллере (паттерн `mvp.http.integration.test.ts`).
- **Frontend** — `api.contract` (`createEisotTestingExport`), e2e-смоук (маршрут/модуль).
- _(Migration test не требуется — миграции нет.)_

## 13. Какие шаблоны входят / не входят (по правилу 4 типов)

- **Тип 4 (импорт/экспорт)** — основная работа: шаблон ростера в ЛКОТ (Excel). Без XML и без обратной загрузки (отложены/не нужны, §4/§8).
- **Тип 1 (печатные документы)** — **не задействован** (C формируется до выдачи документов).
- **Тип 2 (учебные шаблоны)** — только читаем наименование курса как программу; формы программ **не меняем**.
- **Тип 3 (коммуникация)** — вне scope.

## 14. Внешние артефакты (нужны для перехода с provisional на эталон)

1. **Официальный шаблон ростера ЛКОТ** «лица на тестирование» (Excel) — пиннит точные колонки/заголовки/кодировку значений → константа `COLUMNS`.
2. (Опц.) **XML-схема ЛКОТ** — если потребуется XML-сериализация того же ряда (Phase 4 / отложено, §15).
3. (Опц.) **классификатор категорий проверки знаний** с точными наименованиями — если ростер требует строгий код категории, а не свободное наименование программы.

Ключевой артефакт (Excel-шаблон) есть у действующего УЦ либо в кабинете ЛКОТ. Дизайн устойчив к его позднему уточнению: маппинг изолирован в `COLUMNS`.

## 15. Осознанно отложено

- **XML-сериализация / веб-сервис ЛКОТ (live API)** — прямая отправка по ЕСИА/УКЭП (Phase 4).
- **Строгий классификатор категорий проверки знаний** — переиспользование/расширение `lookup.ot_training_programs`; пока наименование программы provisional из `course.title`.
- **Парсер файла-квитанции ЛКОТ** о приёме ростера — если ЛКОТ такой отдаёт.
- **Правило дедупа при мульти-группах** — сейчас «первая отобранная группа»; при необходимости — выбор группы/агрегация работодателя.
