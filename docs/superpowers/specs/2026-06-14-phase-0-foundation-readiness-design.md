## 1. Контекст и реконструкция

Мастер-роадмап писался в предположении **наёмной команды 5-6 человек и бюджета 16-24 млн ₽**. По факту платформу строит владелец совместно с AI-агентом. Это делает значительную часть исходной Phase 0 **неактуальной**, и ряд пунктов уже закрыт кодом иначе, чем планировалось:

| Пункт исходной Phase 0                              | Текущая реальность                                                                                                                                        |
| --------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Yandex Cloud (K8s + PG + S3 + Redis)                | **Заменено и сделано:** один VPS + Docker Compose + Caddy (авто-HTTPS) + MinIO/PG/Redis/RabbitMQ во внутренней docker-сети. Код слит (PR #235/#237/#238). |
| CI/CD pipeline                                      | **Сделано:** GitHub Actions CD (push в `main` → SSH-деплой на сервер) + ночные бэкапы (`infra/backup.sh`).                                                |
| «Собрать команду 5-6 человек», «бюджет ≥22 млн ₽»   | **Неактуально** — solo-владелец + агент. Снято со скоупа.                                                                                                 |
| ЕСИА как критический путь к идентификации (Phase 4) | **Демотировано:** Phase 4 отгружена (#240/#241) на «селфи + паспорт + ручная сверка», ЕСИА-OAuth отложена. ЕСИА теперь — _улучшение_, не блокер.          |

**Вывод:** реальная Phase 0 = **некодовые предпосылки к запуску регулируемой СДО**, в которых доминирует юридическо-компонент (право выдавать значимые документы), а не инфраструктура.

## 2. Критерий «готово» (определение успеха Phase 0)

> **Phase 0 завершена, когда первый реальный пилотный клиент (учебный центр / корпоративный заказчик) может провести настоящую учебную группу на CDOProf по публичному адресу, и выданные удостоверения юридически пригодны для применения.**

Всё, что не служит этой цели напрямую, сознательно откладывается.

## 3. Выбранный подход: A — критический путь к пилоту

Рассмотрены три подхода к последовательности:

- **A. Критический путь к пилоту (ВЫБРАН).** Только минимум для запуска одного пилота: бренд+домен+сервер, один реальный курс, **промежуточная** модель подписи документов; ЕСИА, платежи и полный НЭП — отложены. Срок — недели. Минус: документы пилота на промежуточной подписи, апгрейд до НЭП позже.
- **B. Сначала полное соответствие (отклонён).** Получить НЭП + ЕСИА + реестры + платежи до любого пилота. Срок — месяцы (ЕСИА сам 2-3 мес.). Перестройка ради пилота, медленно.
- **C. Только тех-пилот (отклонён).** Пилот без выдачи юр-документов, вся юр-часть потом. Срок — дни, но не проверяет главную ценность платформы (юридически значимые удостоверения).

**Обоснование выбора A:** по принципу YAGNI он быстрее всех проверяет ядро ценности (поток «обучение → экзамен → удостоверение → реестр»), откладывая медленные гос-процедуры и то, что нужно лишь в Phase 7.

## 5. Направления готовности (workstreams)

Для каждого: _что это · кто делает · чем помогает агент._

### 🟢 На критическом пути к пилоту

**WS1. Бренд + домен**
_Что:_ выбрать название продукта, купить домен, задать адрес отправителя писем (`SMTP_FROM`).
_Зависимости:_ блокирует WS2 (домен в конфиге сервера) и шаблоны удостоверений (бренд на бланке).
_Кто:_ решение владельца.
_Агент:_ генерация вариантов названий, проверка паттернов доступности доменов, рекомендации по `*.ru`/латинице.

**WS2. Запуск сервера (исполнение)**
_Что:_ арендовать VPS (Ubuntu 22.04+, ≥4 ГБ RAM, ~$20-40/мес), направить A-запись DNS, заполнить `infra/.env.production` (секреты, пароли-пары, домен, SMTP), первый `docker compose up --build`, проверка health + TLS, включить CD-секреты GitHub, включить cron бэкапов.
_Зависимости:_ WS1 (домен), SMTP-доступы.
_Кто:_ владелец арендует/даёт SSH-доступ.
_Агент:_ пошаговое сопровождение по [`infra/server-setup.md`](../../../infra/server-setup.md) (10 шагов) + [`infra/bootstrap-admin.md`](../../../infra/bootstrap-admin.md) (первый админ).

**WS3. Модель подписи документов**
_Что:_ решить тип подписи на удостоверениях/протоколах и провайдера. На пилот — **промежуточная простая ЭП + правовая оговорка** (решение B). Зафиксировать план апгрейда до НЭП (провайдер из КриптоПро / СКБ Контур / Тензор; правовое основание: положение об ЭДО, соглашение с обучаемыми).
_Зависимости:_ юридически значимая выдача документов; вход в остаток Phase 6.
_Кто:_ владелец + юрист.
_Агент:_ структурирование решения, перечень требований к провайдеру, чек-лист правового основания. **Юридическую консультацию не даёт.**

**WS4. Пилотный контент**
_Что:_ подготовить ≥1 реальный курс + его документы (шаблон удостоверения) + банк вопросов, готовые к прогону на платформе.
_Зависимости:_ WS2 (платформа живая), бренд для бланка.
_Кто:_ владелец (контент есть из текущей платформы).
_Агент:_ помощь с переносом/настройкой одного курса, шаблона документа и теста.

**WS5. Пилотный клиент**
_Что:_ выбрать 1-2 кандидатов, определить рамку пилота (курс, число слушателей, какие документы), критерии успеха, запасной вариант.
_Зависимости:_ —
_Кто:_ владелец (бизнес-развитие).
_Агент:_ шаблон рамки пилота + критерии отбора кандидата.

### 🟡 Подтвердить, что уже есть (верификация, не «получить»)

**WS6. Образовательная лицензия + программы**
_Что:_ подтвердить наличие действующей образовательной лицензии и зарегистрированных программ, покрывающих тему пилота.
_Кто:_ владелец.
_Агент:_ чек-лист «что проверить».

**WS7. Доступы к гос-реестрам**
_Что:_ получить/подтвердить учётные записи оператора в ФИС ФРДО / ЕИСОТ для реальной сдачи (экспортёры в коде уже готовы — Wave 2).
_Зависимости:_ нужны, когда пилот реально отчитывается в реестр; можно параллельно.
_Кто:_ владелец.
_Агент:_ перечень требуемых доступов, сверка форматов экспорта с эталонами (PROVISIONAL-точки из памяти).

### ⚪ Отложено с критического пути (явно, не «забыто»)

**WS8. ЕСИА ИС-статус** — гос-идентификация. Долгая (2-3 мес.) бюрократическая процедура. Phase 4 работает без неё. Вернуться, когда появится спрос на гос-вход. _(Решение D.)_

**WS9. Платежи (ЮKassa + 54-ФЗ)** — относится к Phase 7. Для B2B-пилота со счётом вручную/бесплатно не нужно. _(Решение E.)_

## 6. Зависимости и критический путь

```
WS1 Бренд+домен ──► WS2 Сервер живой ──► WS4 Курс+документы загружены ──┐
                                                                        │
WS3 Модель подписи (промежуточная) ─────────────────────────────────────┤
                                                                        ▼
                                              Прогон учебной группы ──► Выдача документов ──► (опц.) WS7 Сдача в реестр
WS5 Пилотный клиент ────────────────────────────────────────────────────▲
WS6 Лицензия (подтверждена) ─────────────────────────────────────────────┘

Параллельно/в фоне: WS5, WS6 можно начинать немедленно.
Отложено: WS8 ЕСИА, WS9 Платежи — вне критического пути.
```

**Минимальная цепочка к первому пилоту:**
Бренд+домен → сервер живой → 1 реальный курс + документы → решена модель подписи (хотя бы промежуточная) → прогон группы → выдача документов → _(опц.)_ сдача в реестр.

## 7. Риски и открытые верификации

- **Юридическая значимость документа — главный риск регулируемой СДО.** Промежуточная простая ЭП на пилоте — **сознательный временный компромисс**. Он должен быть помечен как «апгрейд до НЭП перед коммерческим запуском», иначе превращается в скрытый долг. Перед коммерцией — обязательная юр-проверка модели подписи.
- **WS6 (лицензия) — допущение, не факт.** Если лицензия не покрывает тему пилота, критический путь меняется радикально (получение лицензии — месяцы). Подтвердить в первую очередь.
- **WS7 (форматы реестров) PROVISIONAL.** Экспортёры ФРДО/ЕИСОТ/ОТ имеют по 1-4 «swap-точки» до сверки с официальными эталонами. Сдача в реестр на пилоте может потребовать корректировки колонок/формата.

## 8. Вне скоупа Phase 0 (явно)

- Сборка наёмной команды, бюджетные комитеты, оценки 16-24 млн ₽ (solo+AI модель).
- Yandex Cloud / Kubernetes (заменено на VPS+Compose).
- Полный НЭП/КЭП с провайдером (после пилота).
- ЕСИА-интеграция (Phase 4 работает без неё).
- Платежи и 54-ФЗ (Phase 7).
- Миграция всего исторического контента (для пилота — один курс вручную).
