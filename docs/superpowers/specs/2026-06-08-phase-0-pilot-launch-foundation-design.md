## 2. Утверждённый подход — один сервер + Docker Compose (Approach A)

Из трёх рассмотренных подходов (**A** — один сервер + Docker Compose; **B** — управляемые сервисы Yandex Cloud; **C** — гибрид «сервер + управляемая БД») владелец выбрал **A** как переразмеренный под пилот.

**Обоснование A:** дешевле (~2-5к руб/мес), быстрее в настройке, низкорисково (зеркалит локальный `pnpm dev:stack`), полностью скриптуется одним агентом. **Путь миграции** по мере роста: A → C (вынести PostgreSQL в управляемый с PITR-бэкапами) → B (управляемые Redis/Object Storage). Не платим за надёжность, которая на пилоте не нужна.

**Стиль CD — сборка на сервере** (`git pull` → `docker compose build`), без отдельного реестра образов. **Обоснование:** убирает (а) аутентификацию в реестр и (б) трансграничную тягу гигабайтных образов с GitHub-реестра на российский сервер (медленно/нестабильно). Цена — пара минут CPU при деплое, для пилота незаметно. **Апгрейд:** при тяжёлых сборках перейти на реестр (GHCR или Yandex Container Registry) — отдельная итерация.

## 3. Архитектура развёртывания (один сервер)

```
        Интернет → https://<домен>   (наружу открыты только 80/443)
                          │
                    ┌─────▼─────┐
                    │   Caddy   │  авто-HTTPS (Let's Encrypt), reverse-proxy
                    └─────┬─────┘
        внутренняя docker-сеть (порты наружу НЕ публикуются):
   ┌──────────────────────┼────────────────────────────────┐
   │ frontend (Next.js)   :3000   →  /            (кабинеты) │
   │ backend  (NestJS)    :3001   →  /api/v1      (API)      │
   │ realtime (WebSocket) :3002   →  /ws          (push)     │
   │ worker (фоновые задачи, без HTTP-порта)                 │
   │ postgres · redis · rabbitmq · minio · supertokens       │  ← данные в volumes
   └─────────────────────────────────────────────────────────┘
```

**Маршрутизация Caddy** (всё на одном домене → нет CORS, один TLS-сертификат):

| Путь        | Назначение                           |
| ----------- | ------------------------------------ |
| `/`         | frontend `:3000`                     |
| `/api/v1/*` | backend `:3001`                      |
| `/ws`       | realtime `:3002` (WebSocket upgrade) |

> **Важно — `NEXT_PUBLIC_*` пекутся на этапе сборки.** Next.js инлайнит `NEXT_PUBLIC_API_BASE_URL` / `NEXT_PUBLIC_REALTIME_URL` в бандл **при build**, не в рантайме. Значит, образ frontend собирается уже с реальным доменом (`https://<домен>/api/v1`, `wss://<домен>/ws`). Это закладывается в `apps/frontend/Dockerfile` как build-arg, прокидываемый из `docker-compose.prod.yml`.

## 4. Артефакты, которые создаём

1. **`apps/frontend/Dockerfile`** — многоступенчатый прод-образ по образцу backend: `deps` → `build` (Next.js `output: 'standalone'`) → `runtime` (non-root, `node server.js`). Build-arg для `NEXT_PUBLIC_*`.
2. **`infra/docker-compose.prod.yml`** — отдельный от dev прод-стек:
   - добавляет `frontend` и `caddy`;
   - `env_file: .env.production` (не `.env.example`);
   - НЕ публикует порты postgres/redis/rabbitmq/minio/supertokens (только внутренняя сеть);
   - публикует наружу только Caddy (80/443);
   - `restart: unless-stopped` на всех сервисах.
3. **`infra/Caddyfile`** — домен → frontend, `/api/v1/*` → backend, `/ws` → realtime; авто-TLS.
4. **`infra/.env.production.example`** — шаблон с пометками (`# СГЕНЕРИРУЙ: openssl rand -hex 32`, `# ВПИШИ свой домен`), `NODE_ENV=production` (см. §8).
5. **`.github/workflows/deploy.yml`** — триггер на `push` в `main` после зелёного CI; шаги: SSH на сервер (ключ из GitHub Secrets) → `git pull` → `docker compose -f infra/docker-compose.prod.yml up -d --build` → проверка healthcheck.
6. **`infra/backup.sh`** — `pg_dump` (gzip) + `mc mirror` бакета MinIO в каталог бэкапов; retention 14 дней; ставится в host-cron (ночью). Рунбук восстановления — в том же файле.
7. **`infra/server-setup.md`** — разовая настройка: установка Docker/compose, `ufw` (только 22/80/443), не-root пользователь деплоя, клон репозитория, заполнение `.env.production`, первый запуск.
8. **`infra/bootstrap-admin.md`** — создание первого tenant + админ-пользователя в проде (через существующий seed/endpoint), чтобы было кому войти и завести пилот.

## 5. Поток деплоя

**Разово** (по `server-setup.md`): поднять сервер → Docker → firewall → пользователь → клон → заполнить `.env.production` → первый `docker compose -f infra/docker-compose.prod.yml up -d --build` → bootstrap-админ.

**Дальше — автоматически:**

```
push в main → CI (ci.yml: lint+typecheck+contracts+tests+build)
            → зелёный? → deploy.yml:
                 SSH на сервер → git pull
                 → docker compose -f infra/docker-compose.prod.yml up -d --build
                 → backend стартует → авто-накат миграций (DB_MIGRATIONS_ENABLED=true)
                 → healthcheck /api/v1/health/live зелёный
```

> Для владельца это выглядит так: _запушил изменение → через ~3-5 минут оно живёт на домене_. Ручных команд после первичной настройки нет.

## 6. Данные, бэкапы, восстановление

- **Данные** (БД, файлы MinIO, стрим Redis, очереди RabbitMQ) — в именованных docker volumes на сервере; переживают перезапуск контейнеров и `up -d --build`.
- **Бэкап** (`infra/backup.sh`, ночной host-cron): `pg_dump | gzip` → `backups/db-YYYYMMDD.sql.gz`; `mc mirror` бакета → `backups/minio-YYYYMMDD/`; удаление старше 14 дней. Опционально — выгрузка копии в стороннее object storage владельца.
- **Восстановление** — проверенный (не «на бумаге») рунбук: остановить стек → восстановить volume/`psql < dump` → запустить. Репетиция восстановления входит в приёмку (§10).

---

## 7. Безопасность и секреты

- **Сетевой периметр:** наружу только 80/443 (HTTP→HTTPS редирект Caddy). PostgreSQL/Redis/RabbitMQ/MinIO/Supertokens — только во внутренней docker-сети, без `ports:`.
- **Секреты:** генерируются (`openssl rand -hex 32`) и хранятся в `.env.production` **на сервере** (в `.gitignore`, никогда не в git) + дубль для CD в GitHub Secrets (SSH-ключ, host).
- **Обязательные прод-секреты** (валидатор `env:check` обязан их проверять перед стартом): `AUTH_JWT_SECRET`, `SESSION_SECRET`, `INTEGRATION_WEBHOOK_SECRET` (в `.env.example` помечен «обязателен при `NODE_ENV=production`»), `SUPERTOKENS_API_KEY`, `REALTIME_PUBLISH_KEY`, сильные пароли postgres/minio/rabbitmq.
- **Никаких dev-дефолтов в проде:** замена `change-me-in-production`, `postgres/postgres`, `minio/minio123` на сгенерированные значения — обязательный шаг (`server-setup.md`).

## 8. Прод-конфигурация (карта dev → prod)

| Переменная (dev `.env.example`)                                               | Значение в проде                                        | Примечание                                                             |
| ----------------------------------------------------------------------------- | ------------------------------------------------------- | ---------------------------------------------------------------------- |
| `NODE_ENV=development`                                                        | `production`                                            | включает прод-проверки (в т.ч. webhook-secret)                         |
| `DATABASE_URL=...@127.0.0.1:5432`                                             | `...@postgres:5432` + **сильный** пароль                | внутренний docker-хост                                                 |
| `REDIS_URL` / `RABBITMQ_URL` / `S3_*`                                         | внутренние хосты (`redis`/`rabbitmq`/`minio`) + секреты | не публикуются наружу                                                  |
| `AUTH_JWT_SECRET` / `SESSION_SECRET`                                          | `openssl rand -hex 32`                                  | **обязательно** заменить                                               |
| `SUPERTOKENS_API_KEY`                                                         | сгенерировать                                           | должен совпадать с контейнером supertokens                             |
| `SUPERTOKENS_API_DOMAIN` / `WEBSITE_DOMAIN`, `CORS_ORIGIN`, `PUBLIC_BASE_URL` | `https://<домен>`                                       | реальный домен                                                         |
| `NEXT_PUBLIC_API_BASE_URL`                                                    | `https://<домен>/api/v1`                                | **build-time bake** во frontend-образ                                  |
| `NEXT_PUBLIC_REALTIME_URL`                                                    | `wss://<домен>/ws`                                      | build-time bake                                                        |
| `INTEGRATION_WEBHOOK_SECRET`                                                  | сгенерировать (≥10 симв.)                               | обязателен при `NODE_ENV=production`                                   |
| `NOTIFICATIONS_EMAIL_ENABLED=false`                                           | **`true`** + реальные `SMTP_*`                          | **вход по magic-link шлёт письмо — без SMTP пилот не войдёт** (см. §9) |
| `ANTIVIRUS_ENABLED=false`                                                     | **остаётся `false`**                                    | clamav вне объёма пилота (§11)                                         |

## 9. Что требуется от владельца (не может сделать AI-агент)

> **Приоритет владельца (2026-06-08): сначала код.** Все действия из этой таблицы и все юридические/операционные шаги (§11–§13) выполняются **позже**, на этапе исполнения развёртывания. План реализации производит только код- и конфиг-артефакты (§4); **ни одна задача плана не блокируется действием владельца** — деплой на реальный сервер с реальным доменом/SMTP происходит отдельным шагом, когда владелец будет готов.

| Действие                                           | Зачем                                   | Поддержка с моей стороны                      |
| -------------------------------------------------- | --------------------------------------- | --------------------------------------------- |
| Дать SSH-доступ к серверу (IP + ключ деплоя)       | чтобы CD выкатывал                      | пошаговая инструкция в `server-setup.md`      |
| DNS A-запись: `<домен>` → IP сервера               | чтобы домен указывал на сервер          | дам точную инструкцию под регистратора        |
| SMTP для домена (Яндекс 360 / транзакц. провайдер) | вход по magic-link и все письма Phase 5 | инструкция по настройке + заполнению `SMTP_*` |
| Выбрать 2-3 пилотных клиента + их курсы            | кого пускаем первым                     | шаблон выбора + bootstrap данных              |

## 10. Приёмка (как проверяем, что работает)

1. `pnpm env:check` на сервере проходит (все обязательные прод-секреты заданы).
2. `docker compose -f infra/docker-compose.prod.yml ps` — все сервисы `healthy`.
3. `GET https://<домен>/api/v1/health/live` → `200`, TLS-сертификат валиден.
4. **Смоук живого пилота:** открыть `https://<домен>` → запросить magic-link → получить письмо → войти → увидеть кабинет.
5. **Репетиция восстановления:** прогнать `backup.sh`, восстановить дамп в чистый volume, убедиться, что данные на месте.

## 11. Вне объёма (осознанно отложено)

- **ЕСИА** (статус ИС) — длинная юр-процедура; нужна Phase 4, пилоту не блокер. См. §12.
- **НЭП/КЭП-подпись документов** — Phase 6; пилот выдаёт документы без юр-подписи (как и задумано в Phase 1).
- **clamav-демон** — `ANTIVIRUS_ENABLED=false` остаётся; gate на месте, движок включается флагом позже (см. [2026-05-30-v1.1-antivirus-scan-gate-design.md](2026-05-30-v1.1-antivirus-scan-gate-design.md) §9).
- **Управляемые сервисы / Kubernetes / HA / blue-green** — миграция A→C→B по мере роста.
- **Код оплат/вебинаров/прокторинга** — будущие фазы (см. §12), не Phase 0.
- **Мониторинг/алертинг сверх healthcheck+логов** (Sentry, uptime-чек) — лёгкий опциональный задел; полноценно — Phase 11.

## 12. Решения по будущим фазам (зафиксировано на сессии 2026-06-08)

Влияют на последовательность роадмапа, **не на объём Phase 0**; записаны как ориентир для следующих сессий:

- **Phase 4 (идентификация и прокторинг) — код-первый.** Когда дойдём, строим код: WebRTC-видеоидентификация с оператором, селфи+паспорт (загрузка+ручная сверка), прокторинг-запись итогового теста, переключатель метода per-студент, админ-просмотр записи. **ЕСИА-OAuth подключается отдельным включением позже**, когда получен статус ИС. То есть ЕСИА не блокирует старт Phase 4.
- **Pruffme / ЮKassa — договоры считаются заключёнными.** Phase 7 (оплаты: ЮKassa + 54-ФЗ + счета/акты) и Phase 8 (вебинары: Pruffme) разблокированы по контрактной части; их код делаем целиком при подходе фаз.

## 13. Открытые операционные задачи (владелец / исполнение)

1. Поднять/выбрать сервер (если инстанс ещё не создан) и выдать SSH-доступ.
2. Прописать DNS A-запись `<домен>` → IP сервера.
3. Настроить SMTP на домене → `NOTIFICATIONS_EMAIL_ENABLED=true` + `SMTP_*`.
4. Выбрать 2-3 пилотных клиента + курсы для первого запуска.
