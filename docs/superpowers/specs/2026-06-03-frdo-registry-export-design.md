> **⚠️ Ключевое отличие от ОТ-выгрузки.** ОТ-реестр строит строки из **зачислений + протокола проверки знаний**. ФРДО — это выгрузка **по выданным документам об обучении** (удостоверения ПК / дипломы ПП), которые уже существуют в пайплайне документов (`GeneratedDocumentEntity` с `documentNumber`/`documentDate`/`documentType`). Поэтому источник иной (`DocumentsService.listIssuedDocuments`), гранулярность проще (одна строка = один выданный документ), обратной загрузки нет. Остальная форма (durable MVP-модуль, preflight, golden-file `.xlsx`, provisional-маппинг, UI-секция в `gov-export`) переиспользуется из ОТ почти 1:1.

## 1. Назначение и зафиксированные решения

Учебный центр обязан передавать в ФИС ФРДО сведения о выданных документах о дополнительном профессиональном образовании (удостоверения о повышении квалификации, дипломы о профессиональной переподготовке). Сейчас это делает прежняя платформа; чтобы УЦ мог уйти с неё, CDOProf должен **сам формировать корректный файл выгрузки** по выданным документам для загрузки в личном кабинете ФРДО.

Это вторая реализуемая под-цель Волны 2 (B = реестр обученных по ОТ — слита; **A = ФИС ФРДО — этот дизайн**; C = ЕИСОТ «лица на тестирование» — позже). Под-цель B (ОТ) задала прецедент и инфраструктуру.

**Решения, принятые в брейншторме (зафиксированы, одобрены владельцем 2026-06-03):**

1. **Архитектура — durable MVP-модуль, как у ОТ.** Новый модуль `apps/backend/src/modules/mvp/frdo-registry/` зеркалит `ot-registry/`. Пустой `FrdoAdapter` в `integrations/` **не наполняется** — оркестратор `integrations` не персистентен и пишет один summary-item, что не даёт пер-строчных preflight-ошибок (то же отклонение D1, по которому ОТ ушла от адаптера). Адаптерный шов остаётся нетронутым под будущую live-API отправку (Phase 4).
2. **Источник данных — выданные документы**, не зачисления. `DocumentsService.listIssuedDocuments(tenantId, { types: ['certificate','diploma'], status: 'final', from, to })` → `GeneratedDocumentEntity[]`; затем джойн `sourceEntityId`(enrollment) → слушатель (ФИО/СНИЛС/дата рождения/должность) и курс/версия (наименование программы, часы, форма, квалификация).
3. **Гранулярность — одна строка на один выданный документ.** ФРДО отчитывается по документам, не по парам человек×программа. Слушатель с двумя удостоверениями = две строки. Разворот «человек × программа» (как в ОТ) **не нужен**.
4. **Формат — только Excel (`.xlsx`).** Штатная ручная загрузка в ЛК ФРДО — Excel-шаблон. XML-веб-сервис ФРДО требует ЕСИА/УКЭП и статуса ИС — это Phase 4 (live API), а не ручной путь. (ОТ добавляла XML только потому, что у ЛКОТ канонический импорт — XML; у ФРДО ручной канон — Excel.)
5. **Обратной загрузки нет.** ФРДО не возвращает присваиваемые номера — регистрационный номер документа присваивает сам УЦ при выдаче (= наш `documentNumber`). Раунд-трип (как в ОТ) не требуется. Отчёт об ошибках валидации от ФРДО — отложенная возможность (§15).
6. **Provisional-формат — 2 точки-swap'а + баннер.** Официальный Excel-шаблон ФРДО есть у УЦ, но не предоставлен. До сверки всё помечено `PROVISIONAL`; подмена на эталон = правка одной константы на артефакт (классификатор видов документов + колонки `.xlsx`), без изменения логики.

## 2. Что уже готово

**Пайплайн документов** (`apps/backend/src/modules/documents/`) — источник:

- `GeneratedDocumentEntity` (documents.types.ts): `documentType` (`certificate`|`diploma`|`protocol`|`order`|…), `documentNumber` (нумерация, напр. `CERTIFICATE-000001`), `documentDate` (ISO), `status` (`generated`|`final`|`archived`|`revoked`), `sourceEntityType`/`sourceEntityId` (→ enrollment/group), `revokedAt`/`replacesDocumentId` (для исключения аннулированных).
- `DocumentsService.listIssuedDocuments(tenantId, filter)` (documents.service.ts, протестирован — documents.service.test.ts:630) с `IssuedDocumentFilter { from, to, types[], status, groupOrderDocumentId, limit, offset }` — готовая точка перечисления выданных документов.
- `TemplateType` union: `certificate` (удостоверение ПК) и `diploma` (диплом ПП) — ФРДО-отчётные; `protocol`/`order`/прочие — служебные, не выгружаются.
- Нумерация (`NumberingRuleEntity`/`NumberReservationEntity`) → `documentNumber`.

**Данные о слушателе/программе** (`mvp.types.ts`):

- Слушатель: `firstName`/`lastName`/`middleName?` (ФИО), `snils?` (валидатор ПФР), `position?`. **Пробел:** нет даты рождения (§6).
- Версия курса (ProgramMeta, миграция `0030`): `academicHours?`, `trainingType?`, `studyForm?`, `regulatoryBasisCodes?[]`. (Присвоенная квалификация для ПП в схеме явно не выделена — provisional, §4 поля.)

**Инфраструктура регуляторных выгрузок** (прецедент ОТ): модуль `ot-registry/` (service/writer/preflight/rows/dto/controller/tests), права `regulatory.export.read/write` (миграция `0045`), UI-секция в `app/gov-export/page.tsx`, фронт-фича `features/gov-export/` (api/hooks/types + contract-test) — зеркалятся.

**Заглушки, которые ВЫТЕСНЯЮТСЯ (не наполняются):** `FrdoAdapter` (integrations/adapters/frdo.adapter.ts, 8 строк) и общий мастер `gov-export` (provider=`frdo`) — остаются как есть; ФРДО-выгрузка реализуется durable-модулем и отдельной UI-секцией рядом с ОТ.

## 4. Архитектура

**Durable MVP-модуль `frdo-registry/`** (зеркало `ot-registry/`). Вся ФРДО-специфика:

- `frdo-registry.service.ts` (`Scope.REQUEST`) — собирает выданные документы через `DocumentsService.listIssuedDocuments` (зависимость `documents` уже инжектится в MVP — см. конструктор `MvpService(... documents ...)`), джойнит слушателя/курс/версию, строит строки, валидирует (preflight), генерирует `.xlsx`, персистит батч/записи (как ОТ).
- `frdo-registry-rows.ts` — чистая функция `buildFrdoRows(documents, joins)`: одна строка на документ; форматирование дат (`ДД.ММ.ГГГГ`), маппинг вида документа.
- `frdo-registry-preflight.ts` — чистые валидаторы (§9): СНИЛС (ПФР), обязательные поля, исключение аннулированных.
- `frdo-registry-xlsx.writer.ts` — ExcelJS-генератор с `COLUMNS` (единственная точка-swap маппинга поле→колонка, помечена `PROVISIONAL`).
- `frdo-registry.controller.ts` — эндпоинты (§7) под `regulatory.export.read/write`.
- DTO `CreateFrdoRegistryExportDto` (фильтры: период `from`/`to`, типы документов, группа/клиент).
- Типы (`FrdoDocumentKind`, `FrdoRegistryRow`, `FrdoRegistryBatch`, `FrdoRegistryRecord`, `FrdoRegistryExportOutcome`) в `mvp.types.ts`.

Батч/записи регистрируются в request-scoped MVP-state-коллекциях (как ОТ; не забыть зарегистрировать новые коллекции в `mvp-collections.ts` — иначе теряются между HTTP-запросами).

## 5. Альтернативы (отвергнуты)

- **Наполнить `FrdoAdapter`** в `integrations/` — отвергнуто: оркестратор не персистентен, 1 summary-item, нет пер-строчных preflight-ошибок (тот же мотив, что у ОТ — отклонение D1).
- **Источник из зачислений+экзаменов** (как ОТ) — отвергнуто: ФРДО — «по выданным документам»; документы уже несут номер/дату/тип и проходят выдачу/нумерацию/QR.
- **Отдельный модуль `modules/regulatory/`** — отвергнуто: дублирует оркестрацию/UI ОТ; форма MVP-модуля уже доказана.

## 7. Поток выгрузки (outbound)

1. `POST /frdo-registry/exports` с `{ from?, to?, types?: ('certificate'|'diploma')[], groupId?, clientId? }`.
2. Сервис собирает выданные документы (`listIssuedDocuments`, `status:'final'`, не аннулированные), джойнит слушателя (ФИО/СНИЛС/дата рождения/должность), курс/версию (программа/часы/форма/квалификация), конфиг УЦ.
3. **Preflight-валидация** (§9), принцип частичного успеха: валидные строки → файл; невалидные → пер-строчные ошибки на записи; батч → `completed` или `partial_success`.
4. `frdo-registry-xlsx.writer` формирует `.xlsx` по provisional-шаблону → сохраняет в `storage.files`; `batch.fileId` проставлен.
5. Владелец скачивает файл из секции `gov-export`, при необходимости подписывает и загружает в ЛК ФРДО.

Синхронно для разумных партий через текущий жизненный цикл (асинхронный воркер — если потребуется по объёму; решает план).

## 8. Обратная загрузка — отсутствует

ФРДО не присваивает и не возвращает номера записей (рег. номер документа = наш `documentNumber`, присвоен при выдаче). Раунд-трип не реализуется. Отложено (§15): парсер отчёта ФРДО об отклонённых строках, если такой файл понадобится.

## 9. Preflight-валидация

- **Жёстко (строка в ошибки):** есть `documentNumber`; есть `documentDate`; есть ФИО (фамилия+имя); есть наименование программы; вид документа сопоставлен (`frdo_document_kinds`); документ не аннулирован (`revokedAt` пуст, `status='final'`).
- **Предупреждение (строка идёт в файл, но помечается):** отсутствует СНИЛС; отсутствует дата рождения; отсутствуют часы программы. (Provisional — жёсткость поднимется при сверке с эталоном ФРДО.)
- **СНИЛС** при наличии — алгоритм ПФР (переиспользовать из `learners-bulk-import.service.ts`).
- Наименование вида документа эмитится из сида (`exact_name`) — гарантированное совпадение, не из свободного текста.

## 11. Права и безопасность

- Переиспользуются `regulatory.export.read` (чтение/скачивание) и `regulatory.export.write` (формирование) — миграция `0045`.
- **Файл содержит ПДн** (ФИО, СНИЛС, дата рождения, программа) — скачивание под `regulatory.export.read`; учитывать антивирус-гейт скачивания (V1.1) и аудит на формирование/скачивание (`frdo.export_generated`, `frdo.export_downloaded`).
- Дата рождения — ПДн; вводится/хранится под существующими правами редактирования слушателя.

## 12. Тестирование

Трио + специфика (зеркало ОТ):

- **Service unit** — джойн документ→слушатель/курс, одна строка на документ, исключение аннулированных, нормализация дат/СНИЛС, частичный успех.
- **Preflight unit** — табличные кейсы (нет номера/даты/ФИО/программы; несопоставленный вид; аннулирован; битый СНИЛС → warning).
- **Rows unit** — `buildFrdoRows` (маппинг вида документа, формат `ДД.ММ.ГГГГ`).
- **Golden-file `.xlsx`** — фиксирует точный набор/порядок колонок и формат значений (читает структуру обратно).
- **DTO validation** — форма `CreateFrdoRegistryExportDto`.
- **HTTP integration** — границы прав (`regulatory.export.read/write`) на стаб-контроллере (паттерн `mvp.http.integration.test.ts`).
- **Frontend** — `api.contract` (`createFrdoRegistryExport`), e2e-смоук (маршрут/модуль), валидатор даты рождения при импорте.
- **Migration test** — применение `0046`, структура `frdo_document_kinds` + сид, колонка `date_of_birth`.

## 13. Какие шаблоны входят / не входят (по правилу 4 типов)

- **Тип 4 (импорт/экспорт)** — основная работа: шаблон выгрузки в ФРДО (Excel). Без XML и без шаблона обратной загрузки (их нет, §4/§8).
- **Тип 1 (печатные документы)** — **источник** данных (удостоверения/дипломы уже генерируются Pillar A); **не меняем**.
- **Тип 2 (учебные шаблоны)** — только читаем мета-программы (часы/квалификация/форма); формы программ **не меняем**.
- **Тип 3 (коммуникация)** — вне scope.

## 14. Внешние артефакты (нужны для перехода с provisional на эталон)

1. **Официальный `.xlsx`-шаблон ФРДО (ДПО)** из ЛК ФРДО (Рособрнадзор) — пиннит точные колонки/заголовки/кодировку значений → константа `COLUMNS`.
2. **Перечень видов документов ПО/ДПО** с точными кодами/наименованиями ФРДО → сид `lookup.frdo_document_kinds` (swap-1).
3. (Опц.) формат файла-отчёта ФРДО об отклонённых строках — только если потребуется обратный разбор (§15).

Оба ключевых артефакта есть у действующего УЦ либо в ЛК ФРДО. Дизайн устойчив к их позднему уточнению: маппинг изолирован в `COLUMNS`, классификатор — отдельный сид.

## 15. Осознанно отложено

- **XML / веб-сервис ФРДО (live API)** — прямая отправка по ЕСИА/УКЭП (Phase 4).
- **Аннулирование/дубликаты в ФРДО** — отчётность по отозванным/перевыпущенным документам (есть `revokedAt`/`replacesDocumentId`, но отдельный сценарий ФРДО — позже).
- **Парсер отчёта ФРДО** об отклонённых строках — если ЛК ФРДО отдаёт такой файл.
