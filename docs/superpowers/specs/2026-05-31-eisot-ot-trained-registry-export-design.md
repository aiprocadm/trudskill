## 1. Назначение и зафиксированные решения

Учебный центр обязан передавать в реестр Минтруда сведения о лицах, прошедших обучение по охране труда и проверку знаний (ПП №2464). Сейчас это делает прежняя платформа; чтобы УЦ мог уйти с неё, CDOProf должен **сам формировать корректный файл выгрузки** и **принимать обратно регистрационные номера**, которые реестр присваивает каждой записи.

Это первая из трёх под-целей Волны 2 (B = реестр обученных по ОТ; A = ФИС ФРДО; C = ЕИСОТ «лица на тестирование» — обе позже). Выбрана первой, потому что обучение по ОТ — основной ежемесячный обязательный поток УЦ.

**Решения, принятые в брейншторме (зафиксированы, не пересматриваются в этом плане):**

1. **Глубина интеграции — генерация файла, не прямой API.** CDOProf формирует файл; владелец подписывает УКЭП и загружает его в своём ЛКОТ. Прямая отправка в гос-систему требует ЕСИА/УКЭП и статуса ИС — это Phase 4. Адаптерный шов (см. §4) позволит добавить API позже без переделки.
2. **Формат — Excel (официальный шаблон) как основной, XML отложен.** Владелец не знает текущий формат (его делала старая платформа); Excel — штатный способ ручной загрузки и проще для визуальной сверки. Один источник данных, XML добавляется позже как вторая сериализация.
3. **Полный цикл, включая обратную загрузку рег. номеров** — в этом же плане (импорт файла-ответа + проставление номеров в карточку/протокол).
4. **Гранулярность записи — одна строка на (человек × программа).** Каждая запись реестра несёт ровно один ID программы и одно наименование; реестр присваивает рег. номер на каждую программную запись. Комплексный курс, сопоставленный N программам, даёт N строк по слушателю с общими протоколом/датой/результатом.
5. **Сопоставление курс → программа(ы)** редактируется в существующем редакторе мета-программы курса (рядом с `regulatoryBasisCodes`), а непросопоставленные курсы выявляются preflight-валидацией выгрузки. Отдельного CRUD-экрана в V1 нет.

## 2. Что уже готово (≈90%)

**Инфраструктура выгрузок** (`apps/backend/src/modules/integrations/`):

- Адаптерный контракт [`IntegrationAdapter`](../../../apps/backend/src/modules/integrations/adapters/adapter.interface.ts) (7 методов: `prepareExportPayload`, `sendExportBatch`, `handleWebhook`, `mapExternalStatus`, `normalizeError`, `supports`, `testConnection`). Провайдер `eisot` уже зарегистрирован (`ProviderType`), адаптер — пустая заглушка.
- Жизненный цикл задачи (`queued→running→completed/partial_success/failed`), пер-сущностный `ExportItem` с полем **`externalId`** (готовое место под рег. номер), `SyncLog`, dead-letter, идемпотентность, шифрованные `Credential.settingsJsonb`. Персистенция — через `integrations.sync_jobs` / `sync_logs` (+ legacy-вью `legacy_export_tasks`/`legacy_export_items`).
- `ExportTask.resultFileId` — ссылка на сгенерированный файл в `storage.files`.
- UI-мастер [`app/gov-export/page.tsx`](../../../apps/frontend/app/gov-export/page.tsx) подключён к `POST /exports/tasks`, хуки `useExportTasks`/`useSyncLogs`.

**Данные** (большая часть полей реестра уже в схеме):

- Слушатель: `snils` (валидатор ПФР, [`learners-bulk-import.service.ts`](../../../apps/backend/src/modules/mvp/learners-bulk-import.service.ts)), `position` (должность).
- Работодатель: `inn` (10/12 цифр, DTO + DB CHECK — Phase 2 C).
- Программа: `course_versions` мета (`academic_hours`, `training_type`, `study_form`, `learner_category`, `final_assessment_form`, `regulatory_basis_codes[]`, `commission_id`) — миграция `0030`; `lookup.regulatory_acts` уже сидирует `PP_2464_2022`.
- Результат проверки знаний: `ExamResult.passed`; протокол комиссии — выданный документ (Pillar A).

**Единственный реальный пробел в данных:** ID программы в реестре Минтруда (см. §5).

## 3. Поля записи реестра и источники

Одна строка = один человек × одна программа:

| Поле реестра                           | Обяз. | Формат                     | Источник в CDOProf                                  |
| -------------------------------------- | ----- | -------------------------- | --------------------------------------------------- |
| Наименование программы                 | ✔     | строго как в реестре       | `lookup.ot_training_programs.exact_name` (§5)       |
| **ID программы (классификатор)**       | ✔     | число офиц. классификатора | сопоставление курс→программа (§5)                   |
| ФИО                                    | ✔     | текст                      | слушатель                                           |
| СНИЛС                                  | ✔     | `XXX-XXX-XXX YY`           | `learner.snils` (валидатор ПФР)                     |
| Должность                              | ✔     | текст                      | `learner.position`                                  |
| Дата проверки знаний / в удостоверении | ✔     | `ДД.ММ.ГГГГ`               | дата протокола / `ExamResult`                       |
| Номер протокола                        | ✔     | текст                      | протокол комиссии                                   |
| Результат                              | ✔     | удовлетв. / неудовлетв.    | `ExamResult.passed`                                 |
| ИНН работодателя                       | ✔     | 10/12 цифр                 | `client.inn`                                        |
| Сведения об обучающей организации (УЦ) | ✔     | ИНН + рег. № организации   | конфиг тенанта в `eisot`-credential `settingsJsonb` |
| **Рег. номер записи**                  | ↩     | присваивает реестр         | `ExportItem.externalId` (обратная загрузка, §7)     |

> Точный набор колонок/заголовков и кодировка значений (например, как именно пишется «результат») пиннятся по официальному шаблону ЛКОТ — см. §13. Адаптер сопоставляет поля→колонки в одном месте, поэтому уточнение шаблона — drop-in, не передизайн.

## 4. Архитектура

**Подход A — наполнить существующий адаптерный шов** (выбран; альтернативы — отдельный `modules/regulatory/` и переиспользование документного пайплайна — отвергнуты как дублирующие оркестрацию/UI и не подходящие по форме).

Вся регулятор-специфика живёт в:

- `EisotAdapter.prepareExportPayload` — сбор источников + маппинг в поля реестра;
- новый Excel-writer (сервис формирования `.xlsx` по официальному шаблону);
- сид-классификатор `lookup.ot_training_programs` + сопоставление курс→программа;
- новый эндпоинт обратной загрузки (§7).

Оркестратор, таблицы задач/логов, dead-letter, идемпотентность и UF-мастер `gov-export` **не меняются** (только расширяется список `exportType`). Это и есть назначение модуля `integrations`.

## 5. Изменения в данных (3 добавления, без правок исторических файлов)

Номера миграций — следующие свободные (свериться с `apps/backend/migrations/`; на момент дизайна последняя — после `0038`).

1. **`lookup.ot_training_programs`** — глобальный сид официального классификатора программ ОТ (как `regulatory_acts`):
   - `registry_id` (число, ключ классификатора), `exact_name` (строго как в реестре), `program_kind` (`A`|`B`|`V`|`first_aid`|`siz`|`other`), `is_active`.
   - Сид транскрибируется из официального источника ЛКОТ (§13). Известные якоря: ID 1 = «Оказание первой помощи пострадавшим», … ID ~29 = «Безопасные методы и приёмы работ в театрах».
2. **`learning.course_version_ot_programs`** — сопоставление версии курса к **одной или нескольким** программам классификатора (комплексный курс = несколько строк):
   - `(tenant_id, course_version_id, ot_program_registry_id)`, FK на `learning.course_versions` и на `lookup.ot_training_programs(registry_id)`.
3. **Конфиг тенанта** — ИНН и регистрационный номер УЦ как обучающей организации хранятся в `settingsJsonb` активного `eisot`-credential (таблица `integrations.credentials` уже есть). Отдельной таблицы не требуется.

Рег. номер записи переиспользует `ExportItem.externalId` — **новой колонки не нужно.**

## 6. Поток выгрузки (outbound)

1. `POST /exports/tasks` с `{ providerCode:'eisot', exportType:'ot_trained_persons', sourceFilterJsonb:{ groupId | period | clientId } }`.
2. `EisotAdapter.prepareExportPayload` собирает завершённые ОТ-зачисления, у которых есть **сданный** протокол проверки знаний; джойнит слушателя (ФИО/СНИЛС/должность), работодателя (ИНН), мета курса (программа/часы/форма/основание), комиссию/протокол (номер, дата, результат).
3. **Preflight-валидация** (§8). Принцип частичного успеха: валидные строки идут в файл, невалидные — пер-строчные ошибки на `ExportItem`; задача → `completed` или `partial_success`.
4. Разворот «один к многим»: для каждой пары (человек × сопоставленная программа) — отдельная строка.
5. Excel-writer формирует `.xlsx` по официальному шаблону → сохраняет в `storage.files`; `task.resultFileId` проставлен.
6. Владелец скачивает файл из мастера `gov-export`, подписывает УКЭП и загружает в ЛКОТ.

Формирование может выполняться синхронно для разумных партий через текущий жизненный цикл задачи (асинхронный воркер — если потребуется по объёму; решает план по `integration-orchestrator.service.ts`).

## 7. Обратная загрузка рег. номеров (round-trip)

1. После загрузки в ЛКОТ реестр возвращает файл-ответ с присвоенными рег. номерами.
2. `POST /exports/tasks/:id/registry-response` с этим файлом.
3. Сопоставление строк ответа с `ExportItem` по ключу **СНИЛС + номер протокола + ID программы** (устойчиво к перестановке строк).
4. Запись `externalId` (рег. номер) на каждый `ExportItem`; **проставление номера в карточку слушателя и в документ-протокол**; запись аудита `eisot.registration_numbers_imported`.
5. Несопоставленные/конфликтные строки — пер-строчный отчёт, без аборта всей загрузки.

> Точный формат файла-ответа пиннится вместе с шаблоном (§13); парсер устойчив к минорным вариациям колонок.

## 8. Preflight-валидация (чтобы реестр не отклонил)

- **СНИЛС** — алгоритм ПФР (переиспользовать из `learners-bulk-import.service.ts`).
- **ИНН** работодателя — 10/12 цифр.
- Все обязательные поля присутствуют (ФИО, должность, дата, номер протокола, результат).
- **У курса есть сопоставление с программой** классификатора — иначе строка/курс попадает в ошибки с понятной формулировкой «курс X не сопоставлен программе реестра».
- Только записи с `passed` (или явная передача неудовлетворительных — по умолчанию выгружаем сданные; уточнить, нужны ли несданные).
- Наименование программы эмитится из сида (`exact_name`) — гарантированное точное совпадение, не из свободного названия курса.

## 9. UI

- **Мастер `gov-export`** — расширить выбором «Реестр обученных по ОТ (Минтруд)» (`providerCode='eisot'`, `exportType='ot_trained_persons'`) + фильтры (группа / период / клиент). Показывать **список preflight-ошибок до генерации**; кнопка скачивания файла; загрузка файла-ответа реестра.
- **Сопоставление курс→программа(ы)** — поле в существующем редакторе мета-программы курса ([`features/mvp/screens.tsx`](../../../apps/frontend/src/features/mvp/screens.tsx), рядом с `regulatoryBasisCodes`). Навигация — данными в `navigation/model.ts`, без кастомного лейаута.
- Мутации — паттерн `useState`+async (не React Query mutations), `exactOptionalPropertyTypes`.

## 10. Права и безопасность

- **Добавить права `exports.read` / `exports.write`** (миграция + назначение ролям) — сейчас `@Controller('exports')` защищён только `TenantGuard`; выгрузка ПДн в гос-файл требует явного права. Эндпоинт обратной загрузки — `exports.write`.
- Сопоставление курс→программа — под существующим правом редактирования курса.
- **Файл содержит ПДн** (ФИО, СНИЛС, должность) — доступ к скачиванию под `exports.read`; учитывать антивирус-гейт скачивания (V1.1) и аудит на формирование/скачивание/обратную загрузку.

## 11. Тестирование

Трио + специфика:

- **Adapter unit** — маппинг полей, разворот «человек × программа» для комплексного курса, нормализация СНИЛС/даты/результата.
- **DTO validation** — форма запроса `ot_trained_persons` и запроса обратной загрузки.
- **HTTP integration** — границы прав (`exports.read/write`) на стаб-контроллере (паттерн `mvp.http.integration.test.ts`).
- **Golden-file `.xlsx`** — фиксирует точный набор/порядок колонок и формат значений выгрузки.
- **Round-trip** — сопоставление файла-ответа, запись `externalId`, проставление номеров, аудит; устойчивость к перестановке строк и к несопоставленным строкам.
- Preflight — табличные кейсы (битый СНИЛС, нет маппинга, нет должности, неполный протокол).

## 12. Какие шаблоны входят / не входят (по правилу 4 типов)

- **Тип 4 (импорт/экспорт)** — основная работа: шаблон выгрузки в реестр Минтруда (Excel) + шаблон обратной загрузки рег. номеров.
- **Тип 1 (печатные документы)** — источник данных (протоколы/удостоверения уже генерируются Pillar A); **не меняем**.
- **Тип 2 (учебные шаблоны)** — затрагиваются лишь сопоставлением курс→программа; формы программ не меняем.
- **Тип 3 (коммуникация)** — вне scope.

## 13. Внешние артефакты (нужны для реализации)

1. **Официальный `.xlsx`-шаблон ЛКОТ** «Обучение по охране труда» (раздел «Справочная информация» ФГИС СОУТ / экспорт из кабинета ЛКОТ УЦ) — пиннит точные колонки/заголовки/кодировку значений.
2. **Формат файла-ответа** реестра с рег. номерами — для парсера обратной загрузки.
3. **Полный классификатор программ (≈1–29)** с точными наименованиями — для сида `lookup.ot_training_programs` (транскрибировать из официального источника).

Все три есть у действующего УЦ либо доступны в кабинете ЛКОТ. Дизайн устойчив к их позднему уточнению: адаптер изолирует колоночный маппинг, сид — отдельная миграция.

## 14. Осознанно отложено

- **XML-сериализация** той же выгрузки (вторая форма того же payload).
- **Прямая отправка по API ЕСИА/УКЭП** в гос-систему (Phase 4).
- **ФИС ФРДО** (под-цель A) и **ЕИСОТ «лица на тестирование»** (под-цель C) — отдельные планы Волны 2.
- Выгрузка несданных результатов — по умолчанию выгружаем сданные (уточнить при необходимости).
