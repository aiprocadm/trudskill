# Phase 0 — Foundation Readiness Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.
>
> **⚠️ Некодовый план.** Phase 0 — это решения и процедуры **владельца**, не разработка. Классического TDD здесь нет. Исполняемая агентом часть = создать **owner-facing плейбук** (`docs/phase-0/`): чек-листы и шаблоны, превращающие абстрактную готовность в конкретные действия. **Действия владельца** (аренда сервера, выбор бренда, юрист) агент выполнить не может — они идут как чек-боксы внутри плейбука и трекаются владельцем. «Верификация» = подтверждение владельцем, а не прогон тестов.

**Goal:** Создать плейбук готовности Phase 0 — набор из 7 owner-facing документов, проводящих владельца по критическому пути к первому пилоту, выдающему юридически пригодные документы.

**Architecture:** Один индекс (`docs/phase-0/README.md`) + 6 документов-направлений (бренд/домен, запуск сервера, модель подписи, пилотный контент, пилотный клиент, верификация лицензии/реестров). Каждый документ самодостаточен, содержит: цель, что решает владелец, чем помогает агент, чек-лист действий, критерий «готово». Источник истины по содержанию — спек [`2026-06-14-phase-0-foundation-readiness-design.md`](../specs/2026-06-14-phase-0-foundation-readiness-design.md).

**Tech Stack:** Markdown. Деплой-ранбуки уже существуют (`infra/server-setup.md`, `infra/bootstrap-admin.md`) — плейбук на них ссылается, а не дублирует.

## Task 4: WS3 — Модель подписи документов (`docs/phase-0/03-signature-model.md`)

**Files:**

- Create: `docs/phase-0/03-signature-model.md`

- [x] **Step 1: Создать памятку решения по подписи**

> Документ открывается дисклеймером: **«Это структура для решения, а не юридическая консультация. Финальную модель утверждает юрист.»**

Разделы:

- **Цель:** определить, какой подписью заверяются удостоверения/протоколы и каким провайдером.
- **Решение на пилот (из спека §4-B):** промежуточная **простая ЭП + правовая оговорка**; полный НЭП — до коммерческого запуска. Явно отметить как сознательный временный компромисс (спек §7).
- **Сравнение типов подписи** (таблица): простая ЭП / НЭП / КЭП — что это, юр-сила (63-ФЗ), что требуется, когда достаточно. Содержание заполняется конкретно (не «TBD»).
- **Сравнение провайдеров НЭП** (таблица): КриптоПро, СКБ Контур, Тензор — формат интеграции, ориентир по стоимости, что выдают. Помечено «ориентир, уточнить у провайдера».
- **Чек-лист правового основания** (чек-боксы): ☐ положение об ЭДО центра; ☐ соглашение с обучаемыми о признании ЭП; ☐ как QR-проверка соотносится с подписью (модуль QR уже в коде — Pillar A C); ☐ что печатается на бланке про подпись.
- **Чек-лист решения владельца:** ☐ подтвердить промежуточную ЭП на пилот; ☐ выбрать провайдера для НЭП-апгрейда; ☐ задача юристу на правовое основание.
- **Готово когда:** на пилот зафиксирована промежуточная модель + оговорка; выбран провайдер и поставлена задача юристу для апгрейда.
- **Заметка:** при погружении WS3 может стать отдельным спеком (НЭП-интеграция = остаток Phase 6).
