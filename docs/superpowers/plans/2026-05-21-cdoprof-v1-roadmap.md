# CDOProf V1 Implementation Roadmap

> **Назначение документа.** Это **мастер-роадмап** для V1 — высокоуровневая дорожная карта запуска. V1 включает 12 функциональных модулей, поэтому он разделён на 11 фаз. Каждая фаза получит **отдельный детальный план** (с TDD-задачами, файловой структурой, кодом) перед стартом работы по ней.
>
> Этот документ предназначен для **владельца платформы** — чтобы принять решение о старте и финансировании. Для разработчиков планы будут создаваться отдельно по мере подхода каждой фазы.

**Goal:** Запустить пилотную версию CDOProf на собственной платформе с 12 функциональными модулями и 4 кабинетами за 10-14 месяцев работы команды 4-6 человек.

**Architecture:** Параллельная миграция с существующей платформы. Backend и фронтенд развиваются на базе имеющейся кодовой базы (NestJS + Next.js монорепозиторий). Новые модули встраиваются по точкам расширения, заложенным в коде. Фазы максимально параллелятся: бэкенд-команда впереди фронтенд-команды на 1-2 фазы.

**Tech Stack:** TypeScript, NestJS (backend), Next.js (frontend), PostgreSQL, Redis, Yandex Cloud (хостинг + S3), Pruffme (вебинары), ЮKassa (платежи + 54-ФЗ касса), ЕСИА (идентификация).

**Спецификация:** [docs/superpowers/specs/2026-05-21-cdoprof-redesign-design.md](../specs/2026-05-21-cdoprof-redesign-design.md)

## Фазы детально

### Phase 0: Подготовка инфраструктуры и юридическая (4 недели)

**Goal:** Создать готовую инфраструктуру и подготовить юридические основания для всего V1.

**Why first:** Юридические процедуры (ЕСИА, лицензия) занимают 2-3 месяца и **параллельны разработке**. Если запустить их позже — заблокируем фазу 4 (идентификация).

**Deliverables:**

- Arena в Yandex Cloud (Kubernetes + PostgreSQL + Object Storage + Redis)
- CI/CD pipeline для авто-деплоя
- Запущена процедура получения статуса ИС в ЕСИА
- Подписаны договоры с Pruffme, ЮKassa, Yandex Cloud
- Проведена юридическая проверка модели НЭП/КЭП для документов
- Определены пилотные клиенты (2-3 шт.) и пилотные курсы (3-5 шт.)
- Подобран бренд и куплен домен

**Открытые вопросы из спеки, которые решаются здесь:**

- Бюджет проекта (фикс vs T&M)
- Пилотные клиенты и курсы
- Бренд и URL
- Запуск ЕСИА-процедуры

**Tasks (high-level):**

- [ ] Подписать договоры с провайдерами (Yandex/Pruffme/ЮKassa)
- [ ] Развернуть базовую инфраструктуру в Yandex Cloud
- [ ] Настроить CI/CD из существующего репозитория
- [ ] Запустить процедуру ЕСИА (с юристом)
- [ ] Согласовать модель НЭП/КЭП с юристом
- [ ] Определить и согласовать с владельцем 2-3 пилотных клиента
- [ ] Купить домен, настроить DNS

**Estimated effort:** 1 тимлид + 1 DevOps full-time + 0.3 юриста за 4 недели

### Phase 1: Минимальный путь ученика (8 недель)

**Goal:** Ученик может войти, увидеть свои курсы, пройти простой курс с видео, получить документ.

**Deliverables:**

- Магическая ссылка для входа (без пароля)
- Кабинет ученика: главная с "Следующий шаг", список курсов
- Видео-плеер с оглавлением курса
- Простой документ-удостоверение (без юр. подписи пока)
- Базовая дизайн-система: PageHeader, DataTable, кнопки, формы

**Critical Path:** Это база для всех остальных фаз. Без работающего пути ученика нельзя демонстрировать прогресс.

**Tasks (high-level):**

- [x] Магическая ссылка: отправка email с одноразовой ссылкой, верификация
- [x] Авторизация по ссылке, создание сессии
- [x] Кабинет ученика: главный экран с "Следующий шаг"
- [x] Список курсов с прогресс-барами
- [x] Видео-плеер: загрузка видео в S3, потоковая отдача
- [x] Оглавление курса с блокировкой будущих уроков
- [x] Базовый документ: PDF-генерация через `pdfmake` или аналог
- [x] Адаптивность под мобильный (CSS-уровень)

**Estimated effort:** 2 фронт + 1 бэк full-time за 8 недель

**Зависит от:** Phase 0 (инфраструктура)

**Goal:** Сотрудник центра может загрузить Excel с учениками, проверить ошибки, массово назначить курс.

**Deliverables:**

- Раздел "Массовые назначения": drag&drop загрузка Excel

- Партиальное назначение (валидные строки идут в назначение)
- Управление пользователями (CRUD + поиск + фильтры)
- Управление компаниями-клиентами (CRUD)
- Просмотр прогресса по группе/курсу

**Tasks (high-level):**

- [x] Базовый layout админки с навигацией
- [x] Парсер Excel/CSV (универсальный)
- [x] Валидаторы: ФИО, СНИЛС, email, дубликаты
- [x] UI массовой загрузки с предпросмотром ошибок
- [x] Backend: bulk-enrollment API (использовать существующий `BL-003 worker`)
- [x] Worker: фоновая обработка назначений
- [x] Уведомление учеников: email-приглашения
- [x] Раздел "Ученики": список с фильтрами по статусу
- [x] Раздел "Компании": список + карточка компании

**Estimated effort:** 2 фронт + 1 бэк full-time за 8 недель

**Зависит от:** Phase 0 (auth модуль из фазы 1)

### Phase 3: Тестирование и оценивание (6 недель)

**Goal:** Курс может содержать тесты (single/multi/text/essay), система проверяет автоматически, преподаватель проверяет эссе и практические работы.

**Deliverables:**

- Банк вопросов с категориями
- Конструктор тестов (выбор вопросов, рандомизация, таймер)
- Кабинет ученика: прохождение теста с автосохранением
- Автопроверка single/multi/числовых
- Ручная проверка эссе: список к проверке у преподавателя
- Практические задания: загрузка файла, проверка, комментарии
- Журнал результатов теста

**Tasks (high-level):**

- [x] Сущности: Question, QuestionPool, Test, Attempt
- [x] CRUD банка вопросов в кабинете преподавателя
- [x] Конструктор теста (выбор пула, параметры)
- [x] Player теста для ученика
- [x] Автогрейдинг single/multi/number
- [x] UI для ручной проверки эссе
- [x] Загрузка практических работ (файлы в S3)
- [x] Комментарии и циклы доработки

**Estimated effort:** 1 фронт + 1 бэк full-time за 6 недель

**Зависит от:** Phase 1 (учётные записи, кабинет)

### Phase 4: Идентификация и прокторинг (6 недель)

**Goal:** Ученик идентифицирует себя одним из 3 методов перед курсом. Итоговый тест опционально записывается на видео.

**Deliverables:**

- Идентификация через ЕСИА/MAX (если статус ИС получен)
- Видео-идентификация с оператором (видеосвязь через WebRTC)
- Селфи + паспорт (фото + ручная сверка)
- Прокторинг: запись видео во время итогового теста
- Per-student переключатель в админке
- Хранение видео в S3 с автоудалением через срок
- Просмотр видео админом

**Tasks (high-level):**

- [ ] OAuth-интеграция с ЕСИА (требует Phase 0 статуса ИС)
- [ ] WebRTC видео для идентификации с оператором
- [ ] Загрузка селфи и фото паспорта
- [ ] UI выбора метода идентификации
- [ ] UI согласия на видеозапись (152-ФЗ)
- [ ] WebRTC-запись на стороне клиента + загрузка на сервер
- [ ] Бэкенд: хранение в S3 + метаданные в БД
- [ ] Cron-задача автоудаления по сроку
- [ ] UI просмотра записи в админке

**Estimated effort:** 1 фронт + 1 бэк full-time за 6 недель

**Зависит от:** Phase 0 (ЕСИА статус ИС), Phase 1 (кабинет), Phase 3 (тесты)

### Phase 5: Уведомления и переаттестации (4 недели)

**Goal:** Система автоматически шлёт письма по событиям. Через 3 года (или иной срок) до окончания удостоверения ученик и компания получают предложение пройти переаттестацию.

**Deliverables:**

- Email-движок: шаблоны, переменные, рассылка через SMTP/SendGrid/аналог
- События: приглашение на курс, дедлайн через 7 дней, дедлайн сегодня, курс завершён, документ выдан, переаттестация через N дней
- Журнал доставки писем
- Cron-задача проверки сроков переаттестации
- Автоназначение нового курса при подходе срока (по правилам)
- Push-уведомления (PWA)

**Tasks (high-level):**

- [ ] Email-сервис с шаблонами
- [ ] Очередь уведомлений (BullMQ или аналог)
- [ ] Триггеры событий в коде
- [ ] Cron для дедлайнов и переаттестаций
- [ ] Журнал доставки и статусов
- [ ] Push-уведомления в браузере

**Estimated effort:** 1 фронт + 1 бэк full-time за 4 недели

**Зависит от:** Phase 1, Phase 2

### Phase 6: Документы и гос-реестры (8 недель)

**Goal:** Полноценная генерация удостоверений с электронной подписью + автоматическая выгрузка в 4 гос-реестра.

**Deliverables:**

- Конструктор шаблонов документов (HTML/PDF)
- Нумерация удостоверений (по правилам)
- QR-проверка подлинности
- Электронная подпись (НЭП — по решению Phase 0)
- Реестр документов с фильтрами и экспортом
- Аннулирование и перевыпуск
- Автоматическая очередь "к выгрузке" после генерации
- Сборщик XML/Excel для ФРДО
- Сборщик Excel для ЕИСОТ (Минтруд)
- Сборщик для Ростехнадзора
- Сборщик для Минздрава НМО
- Журнал выгрузок с повторами

**Tasks (high-level):**

- [ ] Шаблонизатор документов (HTML + PDF-рендер)
- [ ] Нумерация по правилам (per-курс или per-центр)
- [ ] QR-код с ссылкой на проверку
- [ ] Интеграция с провайдером НЭП (КриптоПро, Тензор, СКБ Контур)
- [ ] Подпись на стороне сервера
- [ ] Раздел "Реестр документов" в админке
- [ ] Очередь "к выгрузке"
- [ ] Сборщики по реестрам (4 шт.)
- [ ] UI гос-экспорта с историей задач

**Estimated effort:** 2 бэк + 1 фронт full-time за 8 недель

**Зависит от:** Phase 1, Phase 3 (логика завершения курса)

**Tasks (high-level):**

- [ ] Интеграция с ЮKassa API
- [ ] Реализация checkout-страницы
- [ ] Webhook-обработчик платежей
- [ ] Чек 54-ФЗ через ЮKassa
- [ ] Шаблоны счёта и акта
- [ ] UI выставления счёта в админке

**Estimated effort:** 1 фронт + 1 бэк full-time за 4 недели

### Phase 9: SCORM-импорт и аналитика (4 недели)

**Goal:** Можно загрузить SCORM-курс от внешнего автора и пройти его. Админ видит дашборды с метриками.

**Deliverables:**

- SCORM 1.2 плеер (через scorm-again или аналог)
- Загрузка SCORM-пакета (zip)
- Регистрация прогресса через SCORM API
- Дашборд админа: completion rate, pass rate, время, drop-off
- Drill-down по компании, курсу, группе

**Tasks (high-level):**

- [x] SCORM-парсер пакета (manifest.xml) — Plan A, `parseScormManifest` (SCORM 1.2; 2004 → unsupported)

- [x] Прогресс через SCORM API — Plan A, cmi commit → завершение materialProgress
- [ ] Дашборд с графиками (Chart.js или Recharts) — **Plan B** (отдельный план)
- [ ] Drill-down фильтры — **Plan B**

**Estimated effort:** 1 фронт + 1 бэк full-time за 4 недели

**Статус:** **Plan A (SCORM-импорт + плеер) реализован** — ветка `feat/2026-06-12-phase-9-plan-a-scorm` (§5.119, миграция 0052). **Plan B (аналитика-дашборд)** — не начат (скоуп в `docs/superpowers/specs/2026-06-12-phase-9-scorm-analytics-design.md` §11).

**Tasks (high-level):**

- [ ] WCAG-аудит (axe-core или Pa11y)

- [ ] UI Excel-конструктора (поля, фильтры, превью)

**Estimated effort:** 2 фронт + 1 бэк full-time за 6 недель

**Зависит от:** Phase 5 (уведомления)

### Phase 11: Финальная подготовка к запуску (3 недели)

**Goal:** Платформа готова принять первого пилотного клиента в проде.

**Deliverables:**

- Нагрузочное тестирование (1000 учеников/мес. capacity)
- Security audit (OWASP Top 10)
- Документация для пилотных клиентов
- Обучение сотрудников центра (методиста, админа)
- Миграция данных 1-го пилотного клиента (если нужно)
- Прод-настройка мониторинга (Sentry, Yandex Cloud Monitoring)
- Backup/restore проверка

**Tasks (high-level):**

- [ ] Нагрузочное тестирование с k6 или Yandex Load
- [ ] Security audit + фиксы

### Сроки (календарные)

При параллельной работе:

- **Фазы 0, 1, 2** идут параллельно (4 + 8 = 8 недель)
- **Фазы 3, 4, 5, 7, 8, 9** идут частично параллельно (с зависимостями)
- **Фазы 6, 10** — после соответствующих
- **Фаза 11** — финальная

Реалистичный календарный срок: **10-14 месяцев**.

| Категория                                     | Сумма (млн руб.) | Комментарий                            |
| --------------------------------------------- | ---------------- | -------------------------------------- |
| Зарплаты (10-14 мес. × 1.7-2.4М/мес.)         | 17-34            | Зависит от состава и сроков            |

> **Замечание о ширине вилки:** 21-43 млн — это очень широкий диапазон. Реальная цифра зависит от 4 факторов: уровень команды (middle vs senior), время на принятие "ваших" 6 решений, сложность миграции данных от текущих клиентов, и неожиданные технические сложности. Для реалистичной оценки нужен бюджетный комитет с конкретной командой.

## Риски и митигация

| Риск                               | Вероятность             | Влияние                      | Митигация                                    |
| ---------------------------------- | ----------------------- | ---------------------------- | -------------------------------------------- |
| ЕСИА статус ИС не получен вовремя  | Средняя                 | Блокирует фазу 4             | Запустить процедуру в Phase 0 (день 1)       |
| НЭП/КЭП — юр. модель не утверждена | Средняя                 | Блокирует фазу 6 (документы) | Юрист в Phase 0                              |
| Команда не собрана                 | Высокая (рынок горячий) | Блокирует всё                | Начать поиск **до** Phase 0                  |
| Пилотный клиент отказался          | Средняя                 | Откладывает запуск           | Подготовить 2-3 кандидата                    |
| Миграция данных сложнее ожидаемого | Средняя                 | +1-2 месяца                  | Phase 0 — оценить структуру старой платформы |
| Бюджет выходит за рамки            | Средняя                 | Полный стоп                  | Контрольные точки каждый квартал             |
| Регулятор ужесточил требования     | Низкая                  | Требует переделок            | Юрист отслеживает изменения                  |
| Большая команда не сработалась     | Низкая                  | -10-20% к скорости           | Хороший тимлид + регулярные ретро            |

## Решение о старте

### Когда можно начинать (минимум для Phase 0)

- [ ] Принят бюджет проекта на сумму не менее **22 млн руб.**
- [ ] Согласован состав команды (хотя бы тимлид + 1 фронт + 1 бэк)
- [ ] Есть кандидат на бренд / название
- [ ] Готов юрист для процедуры ЕСИА и НЭП

### Чего ждать до начала

- [ ] Точного списка пилотных клиентов (можно к концу Phase 0)
- [ ] Финального согласования всех 12 функций V1 (можно до начала фазы 8-9)

### Чего НЕ ждать

- [ ] Идеального ТЗ (его никогда не будет)
- [ ] Получения статуса ИС в ЕСИА (запускается в Phase 0, не блокирует начало)
- [ ] Точного timing-плана до недели (он будет уточняться)
